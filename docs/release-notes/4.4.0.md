# QU.I.R.K. 4.4.0 — Data in Motion

**Released:** 2026-04-29
**Milestone:** v4.4

## What's New

QU.I.R.K. 4.4 closes the "data in motion" gap in the cryptographic inventory: the scanner
now produces a complete picture of how an organization's services negotiate TLS in three
previously-unscanned domains — email protocols, message brokers, and the operational
surface around them. Five new evidence categories flow into the intelligence summary,
the CBOM, and the dashboard, and a sixth named subscore — `data_in_motion` — joins the
existing five (TLS, SSH, API, Identity, Data at Rest) on the executive summary card.

### Email Protocol Scanning

Phase 32 introduced full TLS posture scanning across the standard email protocol surface:
SMTP plaintext (port 25) with STARTTLS-stripping detection, SMTPS (465), submission
(587), IMAP/IMAPS (143/993), and POP3/POP3S (110/995). The scanner uses the existing
sslyze pipeline plus stdlib `smtplib` / `imaplib` / `poplib` fallbacks, so no new
non-core Python dependencies are required for the email path. STARTTLS-stripping
detection on port 25 is emitted as a `MEDIUM`-severity `starttls-downgrade-risk` finding
on every endpoint that successfully negotiates TLS — agentless scanners cannot observe
an active stripping attack, but the *capability* of one is documented as a posture gap.
Weak ciphers on email TLS endpoints surface as `HIGH`-severity findings using the same
algorithm-classification table that drives the rest of the QU.I.R.K. inventory.

### Message Broker TLS

Phase 33 added TLS posture scanning for the three broker types most commonly found in
crypto-relevant data flows. **Apache Kafka** is probed on the standard PLAINTEXT (9092),
SSL (9093), and SASL_SSL (9094) ports; an optional `kafka.admin.AdminClient` enrichment
pulls live broker config when the `quirk[kafka]` extra is installed and credentials are
available, falling back gracefully to sslyze-only TLS posture otherwise. **RabbitMQ** is
probed on AMQP plaintext (5672) and AMQPS (5671), with optional management-API enrichment
on port 15672 (HTTP/HTTPS) using stdlib `urllib` (no `requests` dependency). **Redis**
is probed on plaintext (6379) and TLS (6380) using a raw `ssl.SSLContext.wrap_socket`
with a PING-based plaintext-detection probe; optional `CONFIG GET` enrichment via the
`redis` library distinguishes `AuthenticationError` from `NoPermissionError`. Two cloud
broker surfaces are dispatched as parameters to `scan_rabbitmq_targets`: **Azure Service
Bus** (`namespace.servicebus.windows.net:5671`, AMQPS-only) and **AWS SQS**
(`sqs.<region>.amazonaws.com:443`, HTTPS-only). Plaintext-listener detection on Kafka,
RabbitMQ, and Redis emits `HIGH`-severity findings (`KAFKA-PLAIN`, `AMQP-PLAIN`,
`REDIS-PLAIN`).

### Data-in-Motion Intelligence and Scoring

Phase 34 wired the new email and broker findings into the intelligence summary as six
named `motion_*` counters (e.g., `motion_email_weak_cipher_count`,
`motion_kafka_plaintext_count`) plus five `motion_*_ratio` derivatives. Three motion
ratios feed into the scoring weights, with per-profile multipliers tightening or
relaxing the penalty (`strict` increases, `lenient` decreases). The result lands on
the `data_in_motion` subscore, which joins `tls`, `ssh`, `api`, `identity`, and
`data_at_rest` on the executive summary card. Legacy scans saved before Phase 34 do not
contain motion keys and are detected by their absence — those scans receive full credit
on `data_in_motion` (D-12 backward compatibility), so re-loading a v4.3 scan in 4.4 does
not artificially inflate or deflate the score.

### CBOM Integration

Phase 35 routed email and broker TLS endpoints into the CycloneDX CBOM as Pass-1
algorithm components with quantum-safety classification using the existing
`QUANTUM_SAFETY_MAP` (no map redefinition — D-05). Synthesized-endpoint unit tests
verify the classification pipeline rather than recompute it. Plaintext-only labels
(`KAFKA-PLAIN`, `AMQP-PLAIN`, `REDIS-PLAIN`, `SMTP-STARTTLS`) skip Pass-2 and Pass-3
to prevent hollow certificate entries on endpoints that never negotiated TLS.
`AMQPS/Azure-ServiceBus` passes through the default-TLS branch unchanged. Golden
snapshot fixtures under `tests/test_cbom_motion_golden.py` lock the Pass-1 output
shape for the seven motion-relevant labels.

### Dashboard

Phase 36 added a new `/motion` React route to the dashboard with two surface sections
(email + broker), a per-port TLS-posture grid, plaintext-exposed flags, and STARTTLS
stripping warnings on port 25. The executive summary card grew a sixth row labeled
"Data in Motion" reading the new subscore, and `/api/scan/latest` carries a
`motion_findings` field that the React tab consumes directly. The tab uses the same
shadcn/ui + Tailwind primitives as the rest of the dashboard and degrades gracefully on
scans that contain no motion endpoints (empty state per DASH-05).

## Upgrade Guidance

```bash
pip install --upgrade 'quirk[motion]'
```

The new `[motion]` meta-extra pulls in flat sub-extras: `quirk[email]` (no non-core
deps), `quirk[broker]` (`redis>=5.0`), and `quirk[kafka]` (`kafka-python>=2.0`). Each
remains independently installable if you only need a subset (for example,
`pip install 'quirk[kafka]'` for Kafka-only environments, or `pip install 'quirk[email]'`
for an email-only scan host). The legacy `quirk[redis]` extra is preserved for
back-compat.

Existing v4.3 installations: no breaking changes. Email and broker scanning are off by
default; enable with the `enable_email`, `enable_kafka`, `enable_rabbitmq`, `enable_redis`
config flags (or via the `standard` / `deep` profiles, which now include broker scanning;
the `quick` profile leaves it disabled).

## Chaos Labs

Two new Docker Compose profiles — `email` and `broker` — let you exercise the new
scanners against intentionally-weak local targets without scanning live production:

```bash
docker compose --profile email up -d
docker compose --profile broker up -d
quirk --config <config-with-motion-targets>
```

The email lab runs Postfix + Dovecot with deliberately weak ciphers and a self-signed
cert. The broker lab runs Kafka, RabbitMQ, and Redis with weak TLS configs (DES-CBC3-SHA,
TLS_RSA_WITH_AES_128_CBC_SHA). Expected findings are documented in
`labs/email/expected_results.md` and `labs/broker/expected_results.md`.

## Known Limitations

- **Port 25 cloud egress.** Many cloud providers block outbound SMTP on port 25; a scan
  run from such a host will return `CONNECTION_REFUSED` for that port. The scanner
  handles this gracefully (no crash, finding emitted as the absent-or-blocked category).
- **STARTTLS stripping detection is static.** sslyze cannot observe an active stripping
  attack agentlessly; QU.I.R.K. emits a `starttls-downgrade-risk` MEDIUM on every
  port-25 endpoint that successfully negotiates TLS, on the principle that the
  *capability* is present even when no active attack is observed.
- **KAFKA-04 enrichment is optional.** AdminClient `describe_configs()` requires
  `quirk[kafka]` and authenticated API access. When the library is absent or auth fails,
  the scanner falls back to sslyze-only TLS posture (no functional regression).
- **Azure Service Bus and AWS SQS probes test the TLS layer only.** No SAS-token or
  AWS-credential authentication is required or attempted; the scanner verifies that the
  cloud namespace negotiates TLS but does not exercise queue or topic operations.
- **Phase 36 dashboard motion tab gating** is documented as `wave_0_complete: false`
  pending an unrelated pre-existing SAML scan-window regression (`ISSUE-3` from Phase 24,
  affecting `/api/scan/latest` `identity_findings`). The `/motion` tab itself ships and
  works; the gating-flag deferral is a planning-doc audit-trail concern, not a runtime
  bug. See `.planning/phases/37-gap-closure-and-v4-4-0-release/37-VALIDATION.md`
  "Deferred Gaps" for the full trace.
