## Phase 121: Port-Scope Discovery Control

Addresses a real operator pain point: the hardcoded 6-port TLS list caused GUI scans of the
chaos lab to return zero results because the lab's services bind to high ports.

- **Four port-scope options** on the scan-new form: Common TLS, Top 1000 (default), All ports,
  Custom.
- **`parse_port_spec`** parses nmap-style port specs (singles, ranges, comma lists) with strict
  1–65535 validation, dedup, and sort; custom specs cap at 2048 expanded ports.
- **nmap layer:** `--top-ports 1000` for top1000, `-p-` for all, `-p <csv>` for custom; legacy
  path preserved when unset.
- **Wide-scope hard-fail:** jobs with top1000/all/custom scope that have nmap disabled are rejected
  at submission — no silent 6-port fallback.
- **Zero-result signal:** `GET /api/jobs/{id}/result-summary` returns an explicit "no endpoints
  found" message so operators know the scan ran to completion rather than seeing stale data.
- **Connector suppression:** custom port specs automatically disable the fixed-port email and
  broker connectors that would otherwise override the scope.

### Windows — Unsigned Binary

The Windows sensor zip is **unsigned**. Authenticode code-signing is deferred to a future spike
(requires an EV certificate + CI secret-handling design). Windows SmartScreen may warn on first run.

Workaround: right-click `quirk.exe` → Properties → Unblock, or run `Unblock-File` in PowerShell:
```powershell
Unblock-File -Path .\quirk-windows-sensor\quirk.exe
```

## Upgrade Guidance

```bash

```

**No breaking changes.** All v5.5 configuration files, databases, and scan outputs are
forward-compatible. The port-scope field on `ScanSubmitRequest` is additive (default: `top1000`
preserves prior nmap behavior when nmap is enabled; legacy scans without nmap are unaffected).

Replace the previous `--onedir` build directory with the new zip extract and re-run
`install.ps1` to update the Scheduled Task path if the install location changed.
