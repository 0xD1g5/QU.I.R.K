## Tech Debt / Deferred (non-blocking)

### TD-1 — Shared-token auth model made honest — FIXED in this audit (commit `ba6087e`)
Resolved per user decision. The architecturally-correct fix (given v5.4's locked single-tenant /
no-per-sensor-JWT constraints + one-time-use enrollment tokens) was NOT to wire per-sensor auth, but to
make the shared-token model explicit: `quirk console enroll` no longer implies the minted token is the
push credential (it's labeled provisioning/audit-only); the sensor authenticates with the console's
shared `QUIRK_API_TOKEN` set as `console_api_token`. The distributed lab now runs **auth-ENABLED** with a
shared token (was auth-disabled), demonstrating the authenticated flow. operators-guide §8.1.1 documents
the model. True per-sensor token auth + revocation remains a v5.5 backlog item.

**Original finding (now resolved):** Per-sensor token authentication is not wired —
`quirk console enroll` mints a per-sensor token and stores its SHA-256 hash in `sensor_tokens`, but
`require_auth` validates `POST /api/sensor/push` against the **global** `QUIRK_API_TOKEN` /
`security.api_token`, never against `sensor_tokens`. Consequence: a sensor authenticates only when its
`console_api_token` equals the console's shared token; the distributed-lab e2e relies on auth being
unset. This is consistent with the v5.4 LOCKED constraints (single-tenant, **no per-sensor JWT tokens**,
no per-tenant isolation — 106 D-04), so per-sensor verification was effectively out of scope — BUT the
`quirk console enroll` output is misleading (it prints a token implying per-sensor auth). **Recommended
v5.5 work:** either (a) wire `sensor_tokens` into `require_auth` for true per-sensor auth + revocation,
or (b) change `console enroll` to instruct the operator to use the console's shared `api_token` and
reframe `sensor_tokens` as enrollment-tracking only. Document the shared-token model in the
operators-guide for v5.4.

### TD-2 — sensor_version registry display — FIXED in this audit
`_ingest_envelope` updated `last_push_at` but not `sensor_version`, so the registry always showed "—".
Fixed (commit `680459f`): the push envelope's `sensor_version` is now persisted on each accepted push.
