# Phase 11: Dashboard Wiring Fixes - Research

**Researched:** 2026-04-04
**Domain:** FastAPI dashboard wiring — db_path resolution, env var propagation, CBOM viewer SSH data
**Confidence:** HIGH

## Summary

Phase 11 closes three confirmed integration gaps (GAP-INT-01, GAP-INT-02, GAP-INT-03) found by the
v3.9 milestone audit. All three gaps are surgical one-to-twenty-line fixes in specific, known files.
No architectural changes are required, no new libraries are needed, and no new patterns are introduced
— the fixes use code patterns already present in the codebase.

**Plan 11-01** handles the two wiring issues that break the default E2E flow: (1) the `deps.py`
default db_path disagrees with `config_template.yaml`, causing `/api/scan/latest` to 404 for any
fresh install that uses the generated config; and (2) `server.py` never sets `QUIRK_SERVE_PORT`
before starting uvicorn, so `pdf.py` always targets port 8512 even when serving on a different port.

**Plan 11-02** handles the missing SSH algorithm data in the live dashboard CBOM viewer tab.
`_derive_cbom()` in `scan.py` loops over endpoints but never parses `ep.ssh_audit_json`. The
offline CBOM builder (`builder.py`) already has a correct helper `_extract_ssh_algorithms()` that
can be mirrored exactly.

**Primary recommendation:** Fix all three gaps as isolated, targeted edits in their respective
files. Do not refactor the surrounding code — keep diffs minimal per CLAUDE.md.

## Architecture Patterns

### Pattern 1: db_path Default Resolution (GAP-INT-01)

**What:** The dashboard FastAPI app reads the SQLite DB path from `os.environ.get("QUIRK_DB_PATH", <default>)`. The default value in `deps.py` is `"data/quirk.db"`. The `config_template.yaml` (and thus every file generated by `quirk init`) uses `"./quirk.db"`. These paths resolve differently.

**Root cause (confirmed by audit):**
- `quirk/config_template.yaml:44`: `db_path: "./quirk.db"`
- `quirk/dashboard/api/deps.py:14`: `os.environ.get("QUIRK_DB_PATH", "data/quirk.db")`

**Fix pattern:** Change `deps.py:14` from `"data/quirk.db"` to `"./quirk.db"`. One character change — aligns the fallback default with the config template default. The env var override (`QUIRK_DB_PATH`) continues to work unchanged for non-default setups.

**Code (current broken state):**
```python
# quirk/dashboard/api/deps.py:12-14
def _default_db_path() -> str:
    """Resolve default SQLite path — same logic as run_scan.py."""
    return os.environ.get("QUIRK_DB_PATH", "data/quirk.db")  # WRONG default
```

**Fix:**
```python
def _default_db_path() -> str:
    """Resolve default SQLite path — same logic as run_scan.py."""
    return os.environ.get("QUIRK_DB_PATH", "./quirk.db")  # matches config_template.yaml
```

### Pattern 2: QUIRK_SERVE_PORT Propagation (GAP-INT-02)

**What:** `pdf.py` reads `os.environ.get("QUIRK_SERVE_PORT", "8512")` to build the Playwright URL. `server.py` never sets this env var before calling `uvicorn.run()`. The env var must be set before uvicorn starts so the FastAPI worker process inherits it.

**Root cause (confirmed by audit):**
- `quirk/dashboard/server.py`: `uvicorn.run(...)` called with no env var assignment
- `quirk/dashboard/api/routes/pdf.py:45`: `port = int(os.environ.get("QUIRK_SERVE_PORT", "8512"))`

**Fix pattern:** Add `os.environ["QUIRK_SERVE_PORT"] = str(port)` in `server.py` immediately before the `uvicorn.run()` call. The `port` variable is already in scope at that location.

**Code (current broken state):**
```python
# quirk/dashboard/server.py (line 39 area)
uvicorn.run(
    "quirk.dashboard.api.app:app",
    host=host,
    port=port,
    log_level="info",
)
```

**Fix:**
```python
# Set env var before uvicorn starts so pdf.py worker inherits it
os.environ["QUIRK_SERVE_PORT"] = str(port)
uvicorn.run(
    "quirk.dashboard.api.app:app",
    host=host,
    port=port,
    log_level="info",
)
```

### Pattern 3: SSH Algorithm Parsing in _derive_cbom() (GAP-INT-03)

**What:** `_derive_cbom()` in `scan.py` builds the live dashboard CBOM component list from DB endpoints. It handles TLS (`cert_pubkey_alg`, `tls_version`), JWT (`jwt_scan_json`), and cloud (`cloud_scan_json`) — but not SSH. `ep.ssh_audit_json` is never parsed. SSH endpoints contribute no CBOM components to the dashboard viewer.

**Root cause (confirmed by audit):**
- `quirk/dashboard/api/routes/scan.py::_derive_cbom()` — no `ssh_audit_json` branch
- `quirk/cbom/builder.py::_extract_ssh_algorithms()` — already correctly parses the same data for offline CBOM artifacts

**ssh_audit_json structure (from ssh_scanner.py and builder.py source code):**
```json
{
  "kex": [{"algorithm": "curve25519-sha256", "keysize": null}, ...],
  "key": [{"algorithm": "rsa-sha2-512", "keysize": 3072}, ...],
  "enc": [{"algorithm": "chacha20-poly1305@openssh.com"}, ...],
  "mac": [{"algorithm": "hmac-sha2-256-etm@openssh.com"}, ...]
}
```

Each section is a list of dicts with at minimum an `"algorithm"` key. The `"keysize"` key is optional.

**Fix pattern:** Add an SSH branch inside the `for ep in endpoints` loop in `_derive_cbom()`, mirroring the existing `builder.py::_extract_ssh_algorithms()` logic. The branch should:
1. Check `ep.ssh_audit_json` is not None/empty
2. Parse JSON (wrap in try/except for malformed data)
3. Iterate over sections `("kex", "key", "enc", "mac")`
4. For each entry with an `"algorithm"` key, add to `algo_map` with appropriate type classification

**Builder.py reference implementation (HIGH confidence source):**
```python
# From quirk/cbom/builder.py::_extract_ssh_algorithms() (lines 249-265)
def _extract_ssh_algorithms(ssh_audit_json_str: str | None) -> dict[str, list[dict]]:
    if not ssh_audit_json_str:
        return {}
    try:
        data = json.loads(ssh_audit_json_str)
    except (json.JSONDecodeError, TypeError, ValueError):
        return {}
    result: dict[str, list[dict]] = {}
    for section in ("kex", "key", "enc", "mac"):
        items = data.get(section, [])
        if isinstance(items, list):
            result[section] = items
    return result
```

**Mapping to `_derive_cbom` type/qs fields:**
- `kex` section → `type="key-exchange"` (maps best to key agreement)
- `key` section → `type="signature"` (host key algorithms)
- `enc` section → `type="symmetric-encryption"`
- `mac` section → `type="hash"` (MAC algorithms)

A simpler approach consistent with the existing `_derive_cbom` code: use a single `type="signature"` for all SSH algorithms (same as how JWT algorithms are handled), since `_qs_for_alg` will classify them correctly. The `type` field in `CbomComponent` schema is a free string.

**Alternative section-to-type mapping** (more precise, slightly more code):
```python
_SSH_SECTION_TYPE = {"kex": "key-exchange", "key": "signature", "enc": "cipher", "mac": "hash"}
```

The planner should decide which approach to use. Both are valid for v1.

### Anti-Patterns to Avoid

- **Modifying `_extract_ssh_algorithms` in builder.py**: The fix is in `scan.py::_derive_cbom()` only — do not touch builder.py.
- **Adding new columns or schema changes**: All data is already in `ep.ssh_audit_json`; no migration needed.
- **Refactoring `_derive_cbom()` structure**: Keep the existing loop shape; add the SSH branch as a new `elif ep.protocol == "SSH":` block matching the existing `for json_col in (...)` pattern.
- **Breaking the existing JWT/cloud branch**: The existing `for json_col in (ep.jwt_scan_json, ep.cloud_scan_json)` loop at lines 204-216 handles non-SSH protocols — the SSH branch must be additive, not a replacement.

## Common Pitfalls

### Pitfall 1: CWD-Relative db_path Ambiguity

**What goes wrong:** `"./quirk.db"` and `"data/quirk.db"` are both CWD-relative, so the fix is correct only if the user runs `quirk serve` from the same directory as `quirk --config config.yaml`. This is the expected workflow (both commands run from the project root where `config.yaml` lives).

**Why it happens:** SQLite paths in `deps.py` are relative to the process CWD when uvicorn starts. `quirk serve` is invoked from the project root — same directory where `quirk init` wrote `config.yaml` and where the scan wrote `quirk.db`.

**How to avoid:** The fix is correct for the standard workflow. Add a comment in `deps.py` noting that the path is CWD-relative. Do not introduce `os.path.abspath()` or `pathlib` resolution — this would be a larger change than required.

**Warning signs:** Tests using in-memory DB or fixed test paths will not catch this; verify manually with a live scan.

### Pitfall 2: QUIRK_SERVE_PORT Timing (env var must be set before uvicorn starts)

**What goes wrong:** Setting the env var after `uvicorn.run()` returns is too late. The Playwright PDF route is called by a uvicorn worker thread, not the main process — the env var must be set in the parent process before uvicorn forks/starts workers.

**Why it happens:** `os.environ` is inherited by threads and child processes started after the assignment. `uvicorn.run()` blocks until the server stops — the assignment must appear immediately before the `uvicorn.run()` call.

**How to avoid:** Place `os.environ["QUIRK_SERVE_PORT"] = str(port)` on the line directly before `uvicorn.run(...)`. Do not place it inside the `if not no_open:` block or inside a thread.

**Warning signs:** Tests mock the env var directly; verify with a live non-8512 port test.

### Pitfall 3: ssh_audit_json Vendor Suffix Algorithms

**What goes wrong:** SSH algorithm names in the JSON contain vendor suffixes like `@openssh.com` (e.g., `chacha20-poly1305@openssh.com`). The `classify_algorithm()` function has suffix-stripping logic (Phase 2 decision). If the suffix causes `classify_algorithm` to raise, the `_qs_for_alg` inner function's `except Exception` catches it and returns `"Unknown"` — this is safe behavior.

**Why it happens:** `ssh-audit` reports canonical algorithm names including vendor extensions. The CBOM builder passes these directly to `_register_algorithm` which calls `classify_algorithm`.

**How to avoid:** Use the same `try/except Exception` guard already present in `_qs_for_alg`. Do not add custom stripping logic in `_derive_cbom()` — the classifier already handles this.

### Pitfall 4: Test Environment — pydantic Not Installed (Python 3.14)

**What goes wrong:** Tests that import dashboard routes fail with `ModuleNotFoundError: No module named 'pydantic'` in the Python 3.14 dev environment.

**Why it happens:** pydantic was not installed in the system Python 3.14 environment. Phase 10 audit noted this: "test_gap_closure.py could not run in Python 3.14 environment (pydantic/fastapi broken); verified via static analysis only."

**Current state (verified 2026-04-04):** pydantic 2.12.5 is now installed. `python3 -m pytest tests/test_gap_closure.py tests/test_dashboard_api.py -q` runs and all 11 tests pass.

**How to avoid:** Tests for Phase 11 can be run with `python3 -m pytest`. No workaround needed.

```python
# File: quirk/dashboard/api/routes/scan.py::_derive_cbom()
# Add after the existing TLS/cert_pubkey_alg block and before the jwt/cloud loop:

        # Parse SSH audit JSON for algorithm inventory
        if ep.protocol == "SSH" and ep.ssh_audit_json:
            try:
                ssh_data = json.loads(ep.ssh_audit_json)
            except (json.JSONDecodeError, TypeError, ValueError):
                ssh_data = {}
            _SSH_SECTION_TYPE = {"kex": "key-exchange", "key": "signature", "enc": "cipher", "mac": "hash"}
            for section, alg_type in _SSH_SECTION_TYPE.items():
                for entry in ssh_data.get(section, []):
                    alg = entry.get("algorithm") if isinstance(entry, dict) else None
                    if not alg:
                        continue
                    qs = _qs_for_alg(alg)
                    if alg not in algo_map:
                        algo_map[alg] = {
                            "quantum_safety": qs,
                            "key_size": entry.get("keysize"),
                            "type": alg_type,
                            "sources": set(),
                        }
                    algo_map[alg]["sources"].add(f"{ep.host}:{ep.port}")
```

Note: `_SSH_SECTION_TYPE` can be a module-level constant or defined inline. Inline is fine for ~4 entries.

## Open Questions

1. **`_SSH_SECTION_TYPE` placement — module-level constant vs inline dict**
   - What we know: Both work. The dict has 4 entries and is used only in `_derive_cbom`.
   - What's unclear: Minor style preference.
   - Recommendation: Define inline inside the loop or as a module-level constant with a clear name. Either is acceptable per project conventions (keep diffs minimal).

2. **SSH key_size field in `algo_map` — should it propagate to `CbomComponent`?**
   - What we know: `CbomComponent.key_size` is already an `Optional[int]` field. `ssh-audit` JSON entries may have a `"keysize"` key (integer). `_qs_for_alg` doesn't use key_size.
   - What's unclear: Whether `entry.get("keysize")` can be None or missing for KEX/enc/mac entries (it is set for host key algorithms but often None for others).
   - Recommendation: Pass `entry.get("keysize")` as `key_size` in `algo_map` — safe because it's optional. Existing code at line 194 already does this for TLS cert keys.
