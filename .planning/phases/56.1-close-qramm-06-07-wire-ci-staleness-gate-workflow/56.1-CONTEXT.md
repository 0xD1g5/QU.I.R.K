# Phase 56.1: Close QRAMM-06/07 — Wire CI Staleness Gate Workflow — Context

**Gathered:** 2026-05-08
**Status:** Ready for planning

<domain>
## Phase Boundary

Phase 56.1 closes the documentation/wiring drift left by Phase 56. The
QRAMM staleness pytest gate (`tests/test_qramm_staleness.py`) and the
`quirk qramm status` CLI shipped in Phase 55 (QRAMM-05/06/07), but **no
GitHub Actions workflow ever runs them** — the only existing workflow,
`.github/workflows/dashboard-quality.yml`, is path-filtered to
`src/dashboard/**`. As a result QRAMM-06's "CI pytest gate fails when
`QRAMM_MODEL.last_verified` is more than 90 days old" requirement is
technically unmet on `main`. The same gap exists for the Phase 49 sibling
`tests/test_compliance_freshness.py` (COMPLY-08), which 56.1 also
secures while we are in the area.

**In scope:**
- New `.github/workflows/python-staleness.yml` workflow that runs both
  `tests/test_qramm_staleness.py` and `tests/test_compliance_freshness.py`
  via pytest, plus the QRAMM CLI smoke tests already living in the QRAMM
  staleness file.
- Trigger config: `pull_request` + `push` to main + weekly `schedule` cron
  (Mondays 09:00 UTC). No path filter on PR/push trigger — workflow always
  runs.
- Cadence documentation added to `CLAUDE.md` (a new
  "Staleness Review Cadence" section) and reaffirmed in the
  `quirk/qramm/model_meta.py` docstring.
- REQUIREMENTS.md cross-reference table updated: QRAMM-06 and QRAMM-07
  marked Complete; COMPLY-08 confirmed Complete (already implemented and
  now CI-protected).
- Obsidian phase note + `docs/UAT-SERIES.md` update per the standard
  mandatory phase-completion steps.

**Out of scope:**
- Any change to the staleness threshold (`STALENESS_THRESHOLD_DAYS = 90`
  stays locked).
- Any change to the `QUIRK_CI_STALENESS_OVERRIDE_DATE` env var contract.
- Any change to the QRAMM model itself (`last_verified`, `qramm_version`,
  `source_url`).
- Adding a `CONTRIBUTING.md` or `docs/maintenance.md`.
- Wiring the rest of the pytest suite (e.g., scanner tests, dashboard
  api tests) into CI — that is its own backlog item.
- Notification/alerting (Slack, email) on cron failure — the GitHub
  Actions failure email is sufficient for now.

<decisions>
## Implementation Decisions

### Workflow Scope (D-01)

- **D-01:** A single new workflow file `.github/workflows/python-staleness.yml`
  runs **both** `tests/test_qramm_staleness.py` and
  `tests/test_compliance_freshness.py` in one pytest invocation. The
  workflow name is "Python Staleness Gate". Naming it generically (not
  "QRAMM Staleness") so future model-meta files (e.g., a future PQC
  migration catalog) can be added under the same workflow without
  renaming.
  - Rationale: Both files implement the same staleness pattern from the
    saved feedback memory `feedback_compliance_freshness.md`. Combining
    them costs zero CI time (pytest collects both in <2s) and prevents
    the future "we shipped a third staleness check and it has no CI"
    failure mode that 56.1 is itself fixing.

### Trigger Model (D-02)

- **D-02:** Triggers are `pull_request`, `push: branches: [main]`, and
  `schedule: cron: '0 9 * * 1'` (Mondays 09:00 UTC). The PR/push triggers
  catch model bumps and accidental regressions. The weekly cron catches
  pure calendar drift — a model that ages past 90 days during a quiet
  week with no commits.
  - Rationale: The 90-day window means a quiet 91-day stretch is rare
    but possible during a holiday lull or context-switch. A weekly cron
    bounds that risk to ≤7 days of unflagged staleness without producing
    daily inbox noise.
  - The cron job MUST run the same job as PR — no separate "schedule
    only" steps — so the job logic stays single-sourced.

### Path Filter (D-03)

- **D-03:** No `paths:` filter on the PR/push trigger. The workflow runs
  on every PR and every push to main regardless of which files changed.
  - Rationale: Total workflow runtime is dominated by checkout +
    setup-python (~10–15s). The actual pytest run is <2s and the CLI
    smoke subprocess tests are ~3–5s. The cost of "always run" is
    negligible. The cost of a path-filtered miss — a docs-only or
    dashboard-only PR merging while the model is silently stale — is
    higher because that PR delays the next staleness check by however
    long until the next non-filtered PR.

### Failure Behavior (D-04)

- **D-04:** Workflow fails the build (red ✗) when any of the three
  pytest checks fail:
  1. `test_qramm_model_not_stale` (QRAMM-06 hard gate)
  2. `test_compliance_freshness.py` equivalents (COMPLY-08 hard gate)
  3. `test_qramm_status_cli_smoke_fresh` / `test_qramm_status_cli_smoke_stale_via_override` (QRAMM-07 CLI contract)
  - The CI override env var `QUIRK_CI_STALENESS_OVERRIDE_DATE` is **NOT**
    set in this workflow. Production CI must use the real `today()`. The
    override exists for boundary-condition unit tests only.

### Documentation Home (D-05)

- **D-05:** Cadence documentation lives in two places:
  1. **`CLAUDE.md`** — A new top-level "Staleness Review Cadence" section
     that names both `quirk/qramm/model_meta.py` and `quirk/compliance/`,
     states the 90-day cadence, lists the bump procedure
     (re-verify against `source_url` → bump `last_verified` → commit),
     and references the workflow file. This puts the rule on the same
     enforcement plane as the existing "Chaos Lab Maintenance" and
     "Mandatory Phase Completion Steps" sections.
  2. **`quirk/qramm/model_meta.py`** docstring — keep the existing
     "Per QRAMM-06 — quarterly re-verification cadence" comment;
     reaffirm it points readers to the CLAUDE.md section.
  - Rationale: CLAUDE.md is already the project's enforcement document
    for operational invariants (lab.sh sync, phase completion). Adding a
    new `docs/maintenance.md` would create a parallel docs surface to
    keep in sync; CONTRIBUTING.md doesn't exist and creating it for one
    rule is overkill.

### Requirements Closure (D-06)

- **D-06:** As part of this phase, update
  `.planning/REQUIREMENTS.md`:
  - Mark `QRAMM-06` checkbox `[x]` and update its row in the

  - Mark `QRAMM-07` checkbox `[x]` and update its row from `Pending` to

  - QRAMM-05 is already structurally satisfied; mark `[x]` and table

### Verification Hooks (D-07)

- **D-07:** Verification for 56.1 must include:
  - The workflow file exists and parses (use `actionlint` if available,
    else a yaml.safe_load smoke test in a verifier script).
  - The workflow's `jobs.<job>.steps` includes a step that runs the
    exact pytest command including both test files.
  - `CLAUDE.md` contains the new "Staleness Review Cadence" section
    heading.
  - `REQUIREMENTS.md` shows QRAMM-06 and QRAMM-07 as `[x]`.
  - A dry-run `act` (if installed) or a manual `gh workflow run` is a
    nice-to-have but not required — running the workflow on the PR
    that introduces it is the natural verification.

## Specific Ideas

- **Workflow file name:** `.github/workflows/python-staleness.yml`
  (lowercase, hyphenated, project convention from existing
  `dashboard-quality.yml`).
- **Workflow display name:** `Python Staleness Gate`.
- **Cron expression:** `0 9 * * 1` — Mondays 09:00 UTC (start of week,
  before most US/EU dev hours).
- **Pytest invocation:** `pytest tests/test_qramm_staleness.py
  tests/test_compliance_freshness.py -v` — explicit file paths over
  `-k` matching to keep the gate scope obvious from the CI log.
- **Python version:** `'3.11'` to match the project's stated minimum
  in CLAUDE.md.
- **CLAUDE.md section template (planner may refine):**
  ```
  ## Staleness Review Cadence

  Two project data files carry a `last_verified` date and are gated by
  CI to prevent silent staleness:

  - `quirk/qramm/model_meta.py` — QRAMM model (90-day cadence)
  - `quirk/compliance/__init__.py` — Compliance mappings (365-day cadence)

  When CI fails with a "STALE" assertion:
  1. Re-verify the catalog against its `source_url`.
  2. Bump `last_verified` to today's ISO date in the relevant file.
  3. Commit with message `chore: re-verify <model> catalog (YYYY-MM-DD)`.

  CI workflow: `.github/workflows/python-staleness.yml`
  ```
