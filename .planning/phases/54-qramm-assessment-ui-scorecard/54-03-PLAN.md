    **State B — session exists (auto-resume per D-01):** Show a Card with:
      - Heading `text-lg font-semibold`: "Resume Your Assessment"
      - Body `text-sm text-muted-foreground`: "You have an in-progress assessment. Pick up where you left off."
      - Optional org name + answers_count summary: `{session.org_name ?? 'Untitled assessment'} — {session.answers_count} of 120 answered`
      - Primary Button (variant="default"): "Continue Assessment" → `useNavigate()` to `/qramm/assessment`
      - Secondary Button (variant="outline"): "New Assessment" → toggles inline confirmation
      - Inline confirmation (when toggled) — NO modal — renders below the buttons:
        - Heading `text-base font-semibold`: "Start a New Assessment?"
        - Body `text-sm text-muted-foreground`: "Starting a new assessment will archive your current progress. This cannot be undone."
        - Destructive Button (variant="destructive"): "Confirm New Assessment" → executes the archive flow (see below)
        - Ghost Button (variant="ghost"): "Keep Current Assessment" → hides the inline confirm

    **State C — no session (form):** Single Card with title "QRAMM Org Profile" containing 5 vertically-stacked fields rendered with shadcn Select primitives (and a multi-select for regulatory_obligations — see implementation note below). Submit button at the bottom: variant="default", text "Start Assessment". All option lists come from `qramm-constants.ts`.

    **Submit flow (Start Assessment):**
    1. setLoading(true)
    2. POST `/api/qramm/sessions` with `{ org_name: null }` body (org_name is optional; UI-SPEC does not include an org-name field). Read response, extract `session_id`.
    3. POST `/api/qramm/profiles` with `{ session_id, industry, org_size, geographic_scope, data_sensitivity, regulatory_obligations }`. Read response, extract `profile_id` and `multiplier`.
    4. Update context: `ctx.setSessionId(session_id)`, `ctx.setProfile({ industry, org_size, geographic_scope, data_sensitivity, regulatory_obligations, multiplier })`.
    5. `navigate("/qramm/assessment")`.
    6. On any non-2xx response or network error, set inline error state and surface `text-destructive` text "Could not start assessment — check your connection and try again". Do not navigate.
    7. setLoading(false) in finally.

    **New Assessment flow (Confirm New Assessment):**
    1. If `ctx.sessionId` is non-null: `await fetch('/api/qramm/sessions/{id}', { method: 'DELETE' })`. Ignore non-2xx (still continue — the user wants a clean slate).
    2. `ctx.setSessionId(null); ctx.setProfile(null); ctx.setScoreResult(null); ctx.resetAnswers(new Map())`.
    3. Call the `reload()` returned by `useQRAMMSession()` to re-fetch the session list (which should now be empty or show a different most-recent).
    4. Hide the inline confirm — the page will re-render in State C (form).

    **Regulatory obligations field:** UI-SPEC accepts "multi-select or comma-separated input". Implement as a set of toggleable Badge buttons (one per `REGULATORY_OPTIONS` entry). Click toggles selection. Selected items render with `bg-accent/15 text-accent border-accent` styling; unselected with the default Badge variant. Maintain selected state in local React state as `string[]`. Submit sends the array as-is.

    **A11y:** every Select trigger gets a paired Label (use the new shadcn Label primitive). Label text matches UI-SPEC field names. Confirm button on regulatory toggles uses `aria-pressed={selected}` for screen readers.

    **CSS rules (UI-SPEC):**
    - Use only CSS variables / Tailwind tokens — never hex/hsl literals
    - Card padding: standard shadcn defaults (`p-6` via CardContent is fine)
    - Section spacing: `space-y-6` between Card and inline confirm
    - Use `font-data` only for numeric values (none in this page)

    **Toast for save errors:** UI-SPEC calls for a non-blocking toast. Per RESEARCH.md A1 + Open Question 4, the planner-locked decision is: **DO NOT install sonner in this plan.** Use a minimal inline error banner (`text-sm text-destructive` paragraph below the form's submit button) for now. A future phase may upgrade to sonner.

|----------|-------------|
| User input → form state | Untrusted strings from Select dropdowns and Badge toggles |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-54-12 | Tampering | Form payload | mitigate | All form values come from constrained Select option lists in qramm-constants.ts; freeform text is not accepted in Phase 54 wizard. Backend Pydantic re-validates regardless |
| T-54-13 | Information Disclosure | error message rendering | mitigate | Errors rendered as JSX text — React auto-escapes string children; never use unsafe HTML rendering APIs |
| T-54-14 | Denial of Service | Repeated submit clicks | mitigate | `submitting` state disables the submit button while in flight; UI-SPEC pattern matches executive.tsx handleExportPdf |
| T-54-15 | Repudiation | DELETE on prior session | accept | Single-user tool; user explicitly confirms via "Confirm New Assessment" inline UI; archive is the documented expected behavior per D-02 |
| T-54-16 | Elevation of Privilege | Direct navigation to /qramm/assessment without profile | accept | Plan 04 will display empty-state copy when no session — not a security boundary in a single-user tool |
