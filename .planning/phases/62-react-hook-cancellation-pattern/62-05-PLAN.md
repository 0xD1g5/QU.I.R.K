# Phase 62: React Hook Cancellation Pattern

## Goal

Every data-fetch hook in the dashboard is cancellation-safe — switching scans
mid-fetch never overwrites newer data with stale results, QRAMM debounce coalesces
rapid edits into one request, the auto-fill confirm round-trip preserves the badge
contract, and a CI guard rule prevents future regressions. Closes Pattern C from
the 2026-05-08 audit.

## Requirements Covered

- **HOOK-01** — Cancellation guards in every data-fetch hook (`useScanData`, `useQRAMMSession`, `useTrendsData`, `useScanList`, `useQRAMMPrintData`)
- **HOOK-02** — QRAMM debounce coalescing: rapid answer edits within 300ms POST exactly 1 coalesced batch
- **HOOK-03** — Auto-fill confirm preserves badge removal without full session refetch (`confirmAnswer` method)
- **HOOK-04** — CI guard script (`check-cancelled-guards.sh`) prevents future regressions of the cancellation pattern

## Success Criteria

1. Every hook in `src/dashboard/src/hooks/` gates each `setState` call after an async boundary with an `if (!cancelled)` guard; scan-switch test asserts displayed scan_id always matches the most recently selected scan
2. Typing 20 rapid answer changes within a single 300ms debounce window POSTs exactly 1 coalesced batch to `/api/qramm/assessment/draft`
3. Auto-filling a CVI question and confirming it removes the "Auto-filled from scan" badge without triggering a full QRAMM session refetch
4. `npm run lint:hooks` (and `npm run lint`) flags any unguarded post-await setState in hooks; fires on broken fixture, silent on correct fixture
