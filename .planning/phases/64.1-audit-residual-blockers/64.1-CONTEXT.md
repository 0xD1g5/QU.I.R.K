<domain>
## Phase Boundary

Phase 64.1 is a bridge phase between Phase 64 (Trend Analysis Foundation, human UAT pending) and Phase 65 (Dashboard-Initiated Scan). It has two concrete deliverables:

1. **5 targeted bug fixes** — fix the open BLOCKERs that either directly undermine Phase 64's trend feature or land in Phase 65's exact foundations.
2. **14 formal dispositions** — record a structured `deferred-v4.9` or `wont-fix` block for every remaining open BLOCKER in AUDIT-TASKS.md so no row is left bare.

**The 5 bugs to fix (from the 2026-05-08 audit):**
- `cbom-intel-reports/CR-03` — SOURCE algo hint maps `des→3DES` (should be `des→DES`) and collapses AES variants to `AES-256-GCM`
- `qramm-compliance/BL-03` — `last_verified` staleness uses lexicographic string comparison instead of `datetime.date` arithmetic
- `qramm-compliance/BL-04` — `int(years_raw)` accepts negative/zero values in two interactive entry points
- `cbom-intel-reports/CR-05` — trend session grouping uses second-truncated strftime; two scans within 1 second are merged into one timeline point
- `api-cli-core/CR-08` — `init_db()` calls multiple `_ensure_*` migration functions sequentially with no outer transaction; a failure leaves the DB partially migrated

**Out of scope:** fixing any WARNING or INFO level audit items; changing scan storage model; any dashboard UI work; Phase 65 scan-dispatch wiring.

<decisions>
## Implementation Decisions

### CR-05: Session Window Disambiguation
- **D-01:** Change the strftime format from `"%Y-%m-%d %H:%M:%S"` to `"%Y-%m-%d %H:%M:%f"` in all three session-enumeration functions: `_list_session_timestamps()` and `_list_session_timestamps_n()` in `quirk/dashboard/api/routes/trends.py`, and `_fetch_session_endpoints()` in `quirk/intelligence/trends.py`. Sub-second precision makes same-second scan collisions vanishingly unlikely. **No schema change — uses the existing `scanned_at` column.** The scan_id UUID approach (cleaner semantically but requires schema migration) is deferred to Phase 65 or later if the simple fix proves insufficient.

### CR-03: SOURCE Algo Hint Fix
- **D-02:** In `_extract_algo_from_rule_id()` in `quirk/cbom/builder.py` (lines ~72-78), apply three changes to the ordered `algo_hints` list:
  1. Change `("des", "3DES")` → `("des", "DES")` (DES and 3DES are distinct algorithms)
  2. Add `("aes-256", "AES-256")` and `("aes-128", "AES-128")` entries **before** the generic `"aes"` entry (the ordered-scan invariant requires more-specific patterns first)
  3. Change the catch-all `("aes", "AES-256-GCM")` → `("aes", "AES")` (don't assume variant when the semgrep rule ID doesn't signal one)

### BL-03: Staleness Date Comparison
- **D-03:** In `quirk/compliance/__init__.py:260`, change the lexicographic string comparison `e["last_verified"] < seen[key]["last_verified"]` to use `datetime.date.fromisoformat()` on both sides before comparing. This matches the pattern already used correctly in `quirk/cli/doctor_cmd.py:51-55` and in the CI staleness tests.

### BL-04: Years Validation
- **D-04:** Fix **both** input sites (they're in legacy code but still reachable):
  - `quirk/interactive.py:198` — `int(years_raw) if years_raw else 7`
  - `quirk/assessment/operator_context.py:40` — `int(years_raw) if years_raw else 7`
  - In both places, after parsing, clamp: `if years < 1: raise ValueError(...)` with a clear message. Default (empty input) stays 7 — only explicit non-positive values are rejected.

### CR-08: init_db Transaction Safety
- **D-05:** Wrap the sequential `_ensure_*` calls inside `init_db()` in a single outer try/except so that a mid-migration failure leaves a clear error message rather than silently partial-migrating. Each `_ensure_*` function already uses `with engine.connect() as conn: ... conn.commit()` which is transactional within that function. The risk is cross-function partial migration (e.g., identity columns committed, gcp columns failing). The fix: do not add a redundant outer transaction (SQLite nested transactions are complex); instead, make each individual `_ensure_*` function idempotent and add an integration test that verifies init_db() can be safely called on a partially migrated DB.

### Disposition Framework (14 remaining BLOCKERs)
- **D-06:** Each of the 14 open BLOCKERs that does not get a code fix receives a **structured block** in AUDIT-TASKS.md replacing the bare `[ ] open` status. Format:
  ```
  deferred-v4.9 |
    Rationale: <what the bug is and when it triggers>
    Safe to defer: <why this is safe now — code path, test coverage, user-visible risk>
    Fix phase: <which future phase would naturally address it>
    Risk: <low/medium/high> — <one-line impact statement>
  ```
- **D-07:** Disposition options are `deferred-v4.9` (intend to fix in v4.9) or `wont-fix` (accepted risk or design choice). `wont-fix` requires a higher bar: must state why the behavior is acceptable long-term, not just deferred.

### Plan Structure
- **D-08:** Two plans:
  - **Plan 1** — 5 bug fixes (CR-03, BL-03, BL-04, CR-05, CR-08) + a test for each fix. These land in a single commit so the AUDIT-TASKS.md flip from `[ ] open` → `[x] closed` is part of the same changeset.
  - **Plan 2** — 14 dispositions: read each open BLOCKER, write the structured rationale block in AUDIT-TASKS.md, flip all 14 rows.

</decisions>
