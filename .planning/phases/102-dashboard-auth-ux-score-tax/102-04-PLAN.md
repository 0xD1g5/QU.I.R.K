<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser localStorage → API requests | token stored client-side, injected as X-API-Key on every fetch |
| login form → dashboard API | untrusted token submitted; validated server-side by require_auth (102-02) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-102-12 | Information Disclosure | token in browser history / URL | mitigate | Token entered via `type="password"` input and sent only as the X-API-Key header — never a query param or URL fragment. |
| T-102-13 | Information Disclosure | token in localStorage (XSS surface) | accept | Single-tenant local-use case; accepted per CONTEXT.md scope. No multi-tenant/cookie hardening this milestone. |
| T-102-14 | Spoofing | stale/rotated token still presented by the client | mitigate | 401-while-authenticated triggers logout (per-hook) and the mount probe clears a stale token on 401 → user must re-enter a valid token. |
| T-102-15 | Tampering | auth probe hits an auth-exempt route | mitigate | Probe targets the protected GET /api/scans, NOT /api/health, so auth state is correctly distinguished (Pitfall 6). |
| T-102-16 | Elevation of Privilege | frontend header mismatch with backend | mitigate | Frontend sends X-API-Key matching the 102-02 backend extension; plan depends_on 102-02 to guarantee ordering. |
| T-102-SC | Tampering | npm installs | mitigate | No new npm packages — all shadcn primitives + lucide LogOut already installed (102-UI-SPEC Registry Safety). No slopcheck checkpoint required. |
</threat_model>
