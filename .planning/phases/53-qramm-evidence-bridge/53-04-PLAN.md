```markdown
# Phase 53 — QRAMM Evidence Bridge

## Goal

When a QRAMM assessment session is created, up to 30 CVI dimension questions
are auto-populated with `suggested_answer` values derived from the latest
scan's `CryptoEndpoint` rows — reducing manual assessment effort and
grounding the governance score in live scanner evidence.

## Requirements Covered

- **QRAMM-12** — Bridge auto-populates CVI questions on session create; no `risk_engine` import
- **QRAMM-13** — `suggested_answer` storage with implicit `requires_confirmation`; only confirmed rows score
  - **Schema note:** `requires_confirmation` is NOT a physical column in `qramm_answers`. Confirmation state is derived implicitly from `suggested_answer IS NOT NULL AND answer_value IS NULL` per D-11. QRAMM-13's "requires_confirmation: true" requirement is satisfied by this implicit mechanism — no schema migration needed.
- **QRAMM-14** — Auto-filled rows produce a badge signal (`suggested_answer NOT NULL AND answer_value NULL`); badge disappears on confirmation

## Success Criteria (verified by tests)

1. `POST /api/qramm/sessions` triggers `populate_cvi_suggestions` and the bridge does not import `risk_engine` — verified by `test_bridge_populates_on_session_create` and `test_no_risk_engine_import`
2. Unconfirmed suggestions excluded from score; confirmed suggestions included — verified by `test_unconfirmed_excluded_from_score` and `test_confirmed_included_in_score`
3. RC4-HMAC scan produces lower CVI 1.2 score than AES-256-only scan — verified by `test_rc4_scan_lower_score_than_aes256` (RC4 → 1, AES-256 → 4)

## What Was Built

### 53-01 — Test scaffold (Wave 0 RED)

Created `tests/test_qramm_evidence_bridge.py` with 8 test functions and shared
helpers `_make_bridge_db()` and `_seed_endpoints(db, scenario)`. Scenarios:
`rc4_heavy`, `aes256_only`, `mixed`, `empty`. All tests RED at this stage —
collection-time `ModuleNotFoundError` until Plan 02 ships the module.

### 53-02 — Evidence bridge module

Created `quirk/qramm/evidence_bridge.py` with public function
`populate_cvi_suggestions(session_id, db)` and three private helpers
(`_extract_algorithm_names`, `_parse_json_blob`, `_walk_json_for_alg_strings`).
Implements D-01 SESSION_BRACKET, D-02 silent skip, D-03 multi-field algorithm
extraction, and D-05/D-06/D-07 derivation rules. Imports only
`quirk.cbom.classifier.classify_algorithm` and the two ORM models — zero
risk_engine pull-through. Filters `nist_level is None` to avoid false-vulnerable
inflation (Pitfall 1).

### 53-03 — Router wiring

Modified `quirk/dashboard/api/routes/qramm.py`:
- `create_session`: changed `db.commit()` to `db.flush()` after session insert,
  added a loop to pre-create 30 blank CVI `QRAMMAnswer` rows from
  `QRAMM_QUESTIONS`, then committed once and called `populate_cvi_suggestions`.
- `save_answers`: appended a two-line auto-confirm inside the existing `else`
  branch — `if existing.suggested_answer is not None and item.answer_value is
  not None: existing.confirmed_at = _now_iso()` (D-09).
- `score_session`: untouched per D-10.

### Tests

(no regressions). `test_no_utcnow_in_qramm_module` gate still passes.
```

```markdown
### UAT-Q-53-01 — Evidence bridge auto-populates CVI suggestions on session create

**Goal:** A consultant creating a fresh QRAMM session against a database with
recent scan data sees 30 CVI questions pre-filled with suggestion values.

**Setup:** `quirk serve` against a quirk.db that has `crypto_endpoints` rows
from a recent scan (e.g., the chaos lab `all` profile run).

**Steps:**
1. `curl -X POST http://localhost:8000/api/qramm/sessions -H 'Content-Type: application/json' -d '{"org_name":"UAT-Q-53-01"}'`
2. Note the returned `session_id`.
3. Query rows: `sqlite3 quirk.db "SELECT question_number, suggested_answer, answer_value, evidence_source FROM qramm_answers WHERE session_id = <id> AND dimension = 'CVI' ORDER BY question_number;"`

**Pass criteria:**
- 30 rows returned (Q1–Q30, dimension CVI)
- Every row has `suggested_answer` in {1, 2, 3, 4}
- Every row has `answer_value` NULL
- Every row has `evidence_source` matching `evidence_bridge:scan:YYYY-MM-DD:v1`

### UAT-Q-53-02 — Confirmation flips badge state and updates score

**Goal:** Confirming a suggested answer via `save_answers` updates `confirmed_at`
and includes the row in the maturity score.

**Setup:** A session created per UAT-Q-53-01.

**Steps:**
1. `curl -X POST http://localhost:8000/api/qramm/sessions/<id>/answers -H 'Content-Type: application/json' -d '{"answers":[{"question_number":1,"answer_value":3}]}'`
2. `curl -X POST http://localhost:8000/api/qramm/sessions/<id>/score`

**Pass criteria:**
- After step 1: `sqlite3 quirk.db "SELECT confirmed_at FROM qramm_answers WHERE session_id=<id> AND question_number=1;"` returns a non-NULL UTC timestamp
- After step 2: the score response shows non-zero CVI completion percentage (>0)
- A second sqlite query for unmodified questions confirms their `confirmed_at` remains NULL (badge still visible)
```
