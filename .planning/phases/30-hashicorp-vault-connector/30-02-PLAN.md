<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Vault server (untrusted external) → vault_connector.py | Vault may return malformed data, error responses, or PEM strings of unexpected formats. All API responses must be validated/handled. |
| vault_token / VAULT_TOKEN env → log streams | The token is a sensitive credential. Logging it (even partially) violates V2/Information Disclosure. |
| vault_addr URL → hvac.Client | A malicious or misconfigured vault_addr could trigger SSRF if pointed at internal endpoints. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-30-06 | Information Disclosure | scan_vault_targets / sub-scanners | mitigate | NEVER include token value in any logger.v() / logger.info() / scan_error message; only log presence/absence. The vault-no-token / vault-auth-failed / vault-connection-error strings must contain only error category names, not token contents. |
| T-30-07 | Tampering | _scan_pki_mounts PEM parsing | mitigate | x509.load_pem_x509_certificate() raises ValueError on malformed PEM — caught by the inner try/except per-mount. Malformed PEM does not crash the scanner; it logs and continues to the next mount. |
| T-30-08 | Denial of Service | scan_vault_targets connection | mitigate | hvac.Client(timeout=10) prevents indefinite hangs on unreachable Vault. ConnectionError caught at outer try/except produces vault-connection-error scan_error. Sub-scanner failures isolated via per-resource try/except. |
| T-30-09 | Spoofing | hvac.Client URL validation | accept | vault_addr is developer-supplied via config.yaml or VAULT_ADDR env; not user input from a web request. Plan 03 may add scheme validation if needed; for now accept that misconfiguration is a deployment-time concern, not a runtime exploit. |
| T-30-10 | Elevation of Privilege | Vault token scope | accept | Scanner uses whatever token the operator provides. Documentation in research and Plan 03 vault-seed.sh / config_template.yaml should recommend a least-privilege Vault policy; the code does not enforce token capabilities (Vault does, server-side). |
| T-30-11 | Information Disclosure | dat_scan_json content | mitigate | dat_scan_json must NOT contain the Vault token, the PEM private key (we only have the public cert), or any sensitive Vault data. Verified by inspecting json.dumps payload contents — only mount paths, key names, key types, sig algorithm names, and remediation strings. |
