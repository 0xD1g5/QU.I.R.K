<decisions_made>

- **Library: `google-api-python-client` (NOT `google-cloud-kms`)** — CONTEXT.md D-02 is the canonical decision. STATE.md mentioned google-cloud-kms but that was a roadmap research note, not a locked decision. Research confirmed grpcio conflict is real (google-cloud-kms 3.12.0 hard-requires grpcio>=1.33.2; not installed in QUIRK venv). All three discovery docs (cloudkms.v1.json, sqladmin.v1.json, storage.v1.json) are bundled in google-api-python-client 2.194.0 — no runtime network fetch needed.

- **`DefaultCredentialsError` caught in two places** — at `google.auth.default()` call AND inside each per-resource try/except. GCP_AVAILABLE flag alone does not protect because the error fires at API call time, not import time.

- **`gcs_scan_json` in models.py ORM body + db.py migration guard** — Both needed. ORM for consistency with `cloud_scan_json`; ALTER TABLE for backward-compatible upgrade from v4.2 quirk.db.

- **`session_start` omitted from `scan_gcp_targets()` signature** — Follows existing aws_connector.py/azure_connector.py convention. ISSUE-3 in STATE.md says it's mandatory for all new scanners, but CONTEXT.md Claude's Discretion deferred this to connector implementation style. Advisory: may need to add for Phase 31 trend analysis compatibility.

- **`SSL_MODE_UNSPECIFIED` and null sslMode → HIGH finding** — Same severity as `ALLOW_UNENCRYPTED_AND_ENCRYPTED`. Documented in D-08 and confirmed by research.

- **PQC algorithms in GCP KMS map to existing classifier.py entries** — `ML_KEM_768/1024`, `KEM_XWING`, `PQ_SIGN_ML_DSA_*`, `SLH_DSA_*` all present in GCP KMS v1 API. classifier.py already has ml-kem-*, ml-dsa-*, slh-dsa-* entries. No classifier changes needed.
