# Phase 58 — UI Design Contract

> Visual and interaction contract for the `fetchApi()` API client wrapper and HTTP error
> feedback states introduced by the bearer-token auth, CSRF, and rate-limit middleware.
> Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Scope Note

Phase 58 is a backend security hardening phase. Its frontend surface is narrow but load-bearing:

1. **`fetchApi()` utility** — `src/dashboard/src/lib/api.ts` (new file). Single enforcement
   point for `Authorization: Bearer {token}` and `X-Quirk-Request: 1` header injection on
   every non-GET call. All existing components that call `fetch()` directly must be migrated to
   `fetchApi()`.
2. **HTTP error feedback** — 401, 403 (CSRF), and 429 responses from the new middleware must
   surface human-readable messages in the existing error state pattern, not raw status codes.
3. **Token configuration UX** — No settings UI is introduced in this phase. Token is set via
   environment variable or `config.yaml` (D-01/D-02 in 58-CONTEXT.md). The frontend reads
   the token from a runtime-injected config endpoint or `window.__QUIRK_CONFIG__` — the exact
   mechanism is at executor's discretion. If the token is absent, `fetchApi()` omits the
   `Authorization` header entirely (auth-disabled mode, D-02).

## `fetchApi()` API Client Contract

This is the primary deliverable of Phase 58's frontend work.

### File

`src/dashboard/src/lib/api.ts`

### Signature

```typescript
async function fetchApi(
  url: string,
  options?: RequestInit
): Promise<Response>
```

### Header Injection Rules

| Condition | Header injected | Value |
|-----------|----------------|-------|
| Always | `X-Quirk-Request` | `1` |
| method is POST, PUT, DELETE, PATCH | `Content-Type` | `application/json` (if not already set in `options.headers`) |
| `QUIRK_API_TOKEN` is non-empty (runtime config) | `Authorization` | `Bearer {token}` |
| `QUIRK_API_TOKEN` is empty / absent | `Authorization` | omitted (auth-disabled mode) |

GET calls still pass through `fetchApi()` — it sets `X-Quirk-Request: 1` and
`Authorization` (if token present) but does NOT set `Content-Type`.

### Token Resolution

The token is resolved at module load time from:
1. `window.__QUIRK_CONFIG__.apiToken` (injected by the FastAPI app via a
   `GET /api/config` endpoint or inline script tag in `index.html`)
2. Falls back to empty string if the property is absent or the object is undefined.

The executor may choose either injection mechanism; both satisfy D-08. The `fetchApi()`
function must NOT accept the token as a parameter — it reads from module-level config
resolution only, to prevent token scatter across call sites.

### Migration of Existing `fetch()` Calls

All direct `fetch()` calls in the codebase that target `/api/*` routes must be replaced
with `fetchApi()`. Known call sites as of Phase 58 (source: codebase scan):

| File | Line approx | Method | Notes |
|------|-------------|--------|-------|
| `src/hooks/useScanData.ts` | 27 | GET | Add auth header only |
| `src/hooks/useQRAMMSession.ts` | 28, 48 | GET | Add auth header only |
| `src/hooks/useTrendsData.ts` | 22 | GET | Add auth header only |
| `src/hooks/useQRAMMPrintData.ts` | 29, 51, 52 | GET | Add auth header only |
| `src/hooks/useScanList.ts` | 17 | GET | Add auth header only |
| `src/pages/qramm-profile.tsx` | 53 | DELETE | Full CSRF + auth injection |
| `src/pages/qramm-profile.tsx` | 144, 156 | POST | Full CSRF + auth injection |
| `src/pages/qramm-assessment.tsx` | 98 | GET | Add auth header only |

## HTTP Error Feedback Contract

### 401 Unauthorized

Triggered when: bearer token is set in config but missing or wrong in the request.

| Property | Value |
|----------|-------|
| When shown | `fetchApi()` receives a response with `status === 401` |
| Display location | Inline within the component that initiated the call, replacing the normal content area |
| Component | `EmptyStateCard` with `role="alert"` |
| Heading copy | `Authentication required` |
| Body copy | `The API token is not configured or is incorrect. Set QUIRK_API_TOKEN in your environment and restart the server.` |
| Color | `text-destructive` (existing `--destructive` token) |
| Icon | `lucide-react` `ShieldAlert` at 16px |
| Recovery | No auto-retry. User must fix config and reload. |

### 403 Forbidden (CSRF header missing)

Triggered when: a non-GET request reaches the CSRF middleware without `X-Quirk-Request: 1`.
In normal operation this should NOT occur once `fetchApi()` is deployed — it is a developer
guard, not a user-facing flow. Surface it as a generic API error.

| Property | Value |
|----------|-------|
| When shown | `fetchApi()` receives `status === 403` |
| Display location | Same as 401 — inline, replacing content area |
| Component | `EmptyStateCard` with `role="alert"` |
| Heading copy | `Request blocked` |
| Body copy | `The server rejected this request. If this persists, reload the page or file a bug.` |
| Color | `text-destructive` |
| Icon | `lucide-react` `ShieldAlert` at 16px |
| Recovery | Reload suggestion only. No auto-retry. |

### 429 Too Many Requests

Triggered when: more than 60 mutating requests/min from this IP hit the rate limiter.

| Property | Value |
|----------|-------|
| When shown | `fetchApi()` receives `status === 429` |
| Display location | Inline within the component (same as 401/403) |
| Component | `EmptyStateCard` with `role="alert"` |
| Heading copy | `Too many requests` |
| Body copy | `You've sent too many requests. Wait {N} seconds and try again.` where `{N}` is read from the `Retry-After` response header; if absent, use `60`. |
| Color | `text-destructive` |
| Icon | `lucide-react` `Clock` at 16px |
| Recovery | No auto-retry. Static copy only. No countdown timer. |

### General Non-200 Passthrough

All other non-OK responses continue to use the existing per-hook error pattern:
`setError(\`API error: ${resp.status} ${resp.statusText}\`)` rendered via `EmptyStateCard`.
No change to existing non-200 handling beyond the three cases above.

## Interaction States

### Normal operation (auth disabled or token correct)

`fetchApi()` is transparent. Components behave identically to the current `fetch()` calls.
No visual difference.

### Normal operation (token set, all calls succeed)

Same as above. The injected headers are invisible to the user.

### Auth error (401) — first occurrence on a page load

1. Component mounts, `fetchApi()` fires.
2. 401 response received.
3. Loading spinner is dismissed.
4. `EmptyStateCard` replaces the content area with `ShieldAlert` icon + heading + body.
5. State is sticky — no polling, no retry button.

### Rate-limit hit (429) — during a mutating action

1. User triggers a POST/DELETE/PATCH action (e.g., start QRAMM session, export PDF).
2. 429 response received.
3. The in-progress button returns to idle state (spinner → label).
4. Inline error message appears below the action button: "Too many requests. Wait {N} seconds and try again."
5. No countdown timer is rendered. User reads the static number and waits.
