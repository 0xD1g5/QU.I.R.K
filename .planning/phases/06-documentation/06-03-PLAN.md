```markdown
# AWS Connector Setup

QU.I.R.K.'s AWS connector discovers cryptographic material across ACM certificates, KMS keys, CloudFront distributions, and Elastic Load Balancers.

## Prerequisites

- AWS account with access to the target resources
- boto3 installed: included automatically when you install `quirk` with the `[dashboard]` extra; otherwise `pip install boto3`
- Credentials configured (see [Credential Setup](#credential-setup) below)

## Minimum IAM Policy

Apply this policy to the IAM user or role QU.I.R.K. will use. The permissions map exactly to the boto3 API calls the scanner makes — no wildcards, no write access.

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "QuirkACMReadOnly",
      "Effect": "Allow",
      "Action": [
        "acm:ListCertificates",
        "acm:DescribeCertificate"
      ],
      "Resource": "*"
    },
    {
      "Sid": "QuirkKMSReadOnly",
      "Effect": "Allow",
      "Action": [
        "kms:ListKeys",
        "kms:DescribeKey"
      ],
      "Resource": "*"
    },
    {
      "Sid": "QuirkCloudFrontReadOnly",
      "Effect": "Allow",
      "Action": [
        "cloudfront:ListDistributions"
      ],
      "Resource": "*"
    },
    {
      "Sid": "QuirkELBReadOnly",
      "Effect": "Allow",
      "Action": [
        "elasticloadbalancing:DescribeLoadBalancers",
        "elasticloadbalancing:DescribeListeners"
      ],
      "Resource": "*"
    }
  ]
}
```

## Credential Setup

QU.I.R.K. uses `boto3.Session(region_name=..., profile_name=...)` which resolves credentials via the standard chain:

1. **Environment variables** — `AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`, `AWS_SESSION_TOKEN`
2. **~/.aws/credentials** — named profiles (recommended for local use)
3. **IAM instance role** — for EC2 or ECS execution

For a named profile, add to `~/.aws/credentials`:
```ini
[quirk-readonly]
aws_access_key_id = AKIAIOSFODNN7EXAMPLE
aws_secret_access_key = wJalrXUtnFEMI/K7MDENG/bPxRfiCYEXAMPLEKEY
```

## config.yaml Snippet

```yaml
connectors:
  enable_aws: true
  aws_region: "us-east-1"          # required — set to your target region
  aws_profile: "quirk-readonly"    # optional — uses default credential chain if omitted
```

## What Gets Scanned

| Service | Resource Type | Data Collected |
|---------|--------------|----------------|
| ACM | Certificates | Algorithm, key size, expiry, domain names |
| KMS | Symmetric and asymmetric keys | Key spec (AES_256, RSA_2048, ECC_NIST_P256, etc.) |
| CloudFront | Distributions | TLS policy, associated certificate |
| ELBv2 | Application/Network Load Balancers | TLS listener policy, associated certificate |
```

**docs/connectors/azure.md** structure (per D-12 — lead with RBAC role definition):

```markdown
# Azure Connector Setup

QU.I.R.K.'s Azure connector discovers cryptographic material across Key Vault keys and certificates and App Gateway TLS policies.

## Prerequisites

- Azure subscription with access to the target resources
- azure-identity and azure-keyvault-keys installed: `pip install azure-identity azure-keyvault-keys azure-keyvault-certificates`
- App Gateway scanning also requires: `pip install azure-mgmt-network`
- Service principal or managed identity credentials configured

## RBAC Role Assignment

Assign the following built-in roles to the QU.I.R.K. service principal at the subscription (or resource group) scope:

| Role | Scope | Purpose |
|------|-------|---------|
| `Reader` | Subscription | Required to enumerate resources |
| `Key Vault Reader` | Subscription or per-vault | Read Key Vault metadata |
| `Key Vault Crypto Service Encryption User` | Per vault (if Reader insufficient) | Read key properties |

> **Note:** `Key Vault Reader` is sufficient for most configurations. If you encounter permission errors reading key properties, add `Key Vault Crypto Service Encryption User` to the specific vault.

For App Gateway TLS policy scanning, the service principal also needs:
- `Microsoft.Network/applicationGateways/read` — included in `Reader` role

## Service Principal Credentials

Create a service principal and set these environment variables before running a scan:

```bash
export AZURE_CLIENT_ID="00000000-0000-0000-0000-000000000000"
export AZURE_TENANT_ID="00000000-0000-0000-0000-000000000000"
export AZURE_CLIENT_SECRET="your-client-secret"
```

QU.I.R.K. uses `DefaultAzureCredential()` which resolves via the standard chain:
1. Environment variables (above) — for service principals
2. Managed identity — for Azure VM or App Service execution
3. Azure CLI (`az login`) — for local development

## config.yaml Snippet

```yaml
connectors:
  enable_azure: true
  azure_subscription_id: "00000000-0000-0000-0000-000000000000"  # required
  azure_keyvault_urls:
    - "https://myvault.vault.azure.net"
    - "https://anothervault.vault.azure.net"
```

> **Note:** `azure-mgmt-network` is imported conditionally. If not installed, App Gateway scanning is skipped silently. Install it explicitly to enable App Gateway TLS policy discovery.

## What Gets Scanned

| Service | Resource Type | Data Collected |
|---------|--------------|----------------|
| Key Vault | Keys | Key type (RSA, EC, oct), key size, enabled/disabled state |
| Key Vault | Certificates | Algorithm, expiry, issuer |
| App Gateway | TLS policies | Cipher suite policy, minimum TLS version |

**docs/connectors/docker.md** structure (per D-13):

```markdown
# Docker Container Connector Setup

QU.I.R.K.'s container connector uses Syft to enumerate crypto libraries embedded in Docker container images. It identifies outdated crypto libraries (e.g., openssl 1.0.x, PyCryptodome 2.x) that carry known vulnerabilities or deprecated algorithm support.

## Prerequisites

- Docker installed and running
- Syft installed: `pip install syft` or follow [Syft installation docs](https://github.com/anchore/syft)
- Docker socket accessible at `/var/run/docker.sock` (default on macOS and Linux)

> **Note:** On macOS with Docker Desktop, the socket is at `/var/run/docker.sock` by default. On Linux, you may need to add your user to the `docker` group: `sudo usermod -aG docker $USER && newgrp docker`

## config.yaml Snippet

```yaml
connectors:
  enable_container: true
  container_targets:
    - "nginx:1.18"                    # public registry image by tag
    - "myregistry.example.com/app:v2" # private registry image
    - "sha256:abc123..."              # image by digest
```

## What Gets Scanned

Syft enumerates the software bill of materials (SBOM) from the image layers. QU.I.R.K. filters for crypto-relevant packages:

| Library Pattern | Finding Type |
|----------------|--------------|
| `openssl` < 1.1.1 | `OUTDATED_CRYPTO_LIB` |
| `libssl` < 1.1.0 | `OUTDATED_CRYPTO_LIB` |
| `cryptography` (Python) < 3.0 | `OUTDATED_CRYPTO_LIB` |
| `pyopenssl` < 20.0 | `OUTDATED_CRYPTO_LIB` |
| `pycryptodome` / `pycrypto` | `OUTDATED_CRYPTO_LIB` |

## Private Registry Access

For private registries, authenticate before running the scan:
```bash
docker login myregistry.example.com
```
QU.I.R.K. inherits Docker credential helpers from the host environment.
```

**docs/connectors/git.md** structure (per D-13):

```markdown
# Git / Source Code Connector Setup

QU.I.R.K.'s source code connector uses semgrep to scan Git repositories for cryptographic anti-patterns: hardcoded keys, weak algorithm usage, deprecated protocol invocations, and insecure random number generation.

## Prerequisites

- semgrep installed: `pip install semgrep` or `brew install semgrep` on macOS
- Git installed (for remote repo cloning)
- Repository accessible locally or via HTTP/SSH

## config.yaml Snippet

```yaml
connectors:
  enable_source: true
  source_targets:
    - "/path/to/local/repo"           # local filesystem path
    - "https://github.com/org/repo"   # public GitHub repo (cloned temporarily)
    - "http://gitea.internal/org/repo" # internal Gitea instance
```

## What Gets Scanned

semgrep runs QU.I.R.K.'s built-in crypto anti-pattern ruleset against all source files. Detected patterns:

| Anti-Pattern | Finding Type | Example |
|-------------|--------------|---------|
| Weak algorithm usage | `WEAK_ALGORITHM` | `hashlib.md5()`, `DES.new()` |
| Hardcoded cryptographic key | `HARDCODED_KEY` | `key = "mysecretkey123"` |
| Weak random for crypto | `WEAK_RANDOM` | `random.randint()` for key generation |
| Deprecated protocol | `DEPRECATED_PROTOCOL` | `ssl.PROTOCOL_TLSv1` |

## Supported Languages

semgrep rules cover: Python, Go, Java, JavaScript/TypeScript, Ruby.

## Private Repository Access

For SSH-authenticated repos, ensure your SSH key is loaded:
```bash
ssh-add ~/.ssh/id_ed25519
```

For token-authenticated HTTP repos (Gitea, GitHub), the URL can include credentials:
```yaml
source_targets:
  - "http://admin:token@gitea.internal/org/repo"
```
Or use a `.netrc` file for credential storage without embedding tokens in config.yaml.
