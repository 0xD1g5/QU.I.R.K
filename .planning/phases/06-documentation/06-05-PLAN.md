**Document structure:**

### Section 1: What Is a CBOM? (audience: compliance officers)

Explanation in plain English, no jargon:

A Cryptographic Bill of Materials (CBOM) is a machine-readable inventory of every cryptographic algorithm, key, and protocol in use across a system or network. It is the cryptographic equivalent of a software bill of materials (SBOM) — a list of ingredients, not just a report that says "encryption is present."

**Why it matters for compliance:**
- NIST SP 800-208 (Recommendation for Stateful Hash-Based Signature Schemes) and the broader NIST Post-Quantum Cryptography (PQC) program require organizations to inventory their cryptographic assets before beginning migration
- CNSA 2.0 (Commercial National Security Algorithm Suite 2.0, August 2022) sets migration deadlines for NSS and national security systems — a CBOM is the starting artifact for CNSA 2.0 compliance work
- NIST IR 8547 and the emerging NIST CBOM guidance explicitly use the CycloneDX CBOM schema that QU.I.R.K. produces

**What a CBOM contains:**
Each component entry in the CBOM represents one discovered cryptographic algorithm and includes:
- The algorithm name (e.g., `RSA-2048`, `TLS_AES_256_GCM_SHA384`, `ssh-ed25519`)
- Where it was found (the system, service, or endpoint)
- Its quantum-safety classification (`quantum-safe`, `quantum-vulnerable`, or `unknown`)
- The NIST PQC security level (0–5 scale)

**CBOM output formats:** QU.I.R.K. produces two artifacts per scan:
- `cbom-{timestamp}.cdx.json` — CycloneDX 1.6 JSON format
- `cbom-{timestamp}.cdx.xml` — CycloneDX 1.6 XML format

Both files validate against the CycloneDX 1.4+ schema.



### Section 2: How QU.I.R.K. Produces the CBOM (audience: technical readers, consultants)

**The pipeline — five steps:**

1. **Discovery** — QU.I.R.K. scans TLS/SSH/JWT/container/cloud targets and produces `CryptoEndpoint` records. Each endpoint has the raw cryptographic material: cipher suites, certificate algorithm, SSH KEX algorithms, JWT signing algorithm, etc.

2. **Algorithm extraction** — The CBOM builder (`quirk/cbom/builder.py`) reads each `CryptoEndpoint` and extracts individual algorithm names. One endpoint typically contributes multiple algorithms (e.g., a TLS endpoint yields: certificate key algorithm + accepted cipher suites + EC curves).

3. **Classification** — `classify_algorithm()` in `quirk/cbom/classifier.py` maps each algorithm name to a `(CryptoPrimitive, nist_level, classical_bits)` tuple:
   - `CryptoPrimitive`: `ASYMMETRIC_ENCRYPTION`, `DIGITAL_SIGNATURE`, `KEY_AGREEMENT`, `HASH`, `MAC`, `SYMMETRIC_ENCRYPTION`, etc.
   - `nist_level`: NIST PQC security level (0 = quantum-vulnerable, 1–5 = quantum-safe at increasing strength)
   - `classical_bits`: equivalent classical security strength

4. **Quantum safety labeling** — `quantum_safety_label(nist_level)` assigns the human-readable label:
   - `nist_level == 0` → `"quantum-vulnerable"` (broken by Shor's or Grover-weakened below 128-bit effective security)
   - `nist_level >= 1` → `"quantum-safe"` (meets NIST PQC security level 1 or higher)
   - `nist_level is None` → `"unknown"` (algorithm not recognized or no algorithm present)

5. **CycloneDX serialization** — Deduplicated algorithm components are written to a `cyclonedx.model.bom.Bom` object and serialized to JSON and XML using the CycloneDX Python library (`cyclonedx-python-lib`).

**Algorithm classification examples (documented for consultant accuracy):**

| Algorithm | Primitive | NIST Level | Label | Reason |
|-----------|-----------|-----------|-------|--------|
| RSA-2048 | ASYMMETRIC_ENCRYPTION | 0 | quantum-vulnerable | Shor's algorithm solves integer factorization in polynomial time |
| ECDSA-P256 | DIGITAL_SIGNATURE | 0 | quantum-vulnerable | Shor's algorithm solves elliptic curve discrete log |
| AES-256-GCM | SYMMETRIC_ENCRYPTION | 1 | quantum-safe | Grover halving leaves 128-bit effective security — above threshold |
| AES-128 | SYMMETRIC_ENCRYPTION | 1 | quantum-safe | Grover halving leaves 64-bit effective — marginal but classified safe |
| SHA-256 | HASH | 0 | quantum-vulnerable | Grover halving to 128-bit effective — below NIST threshold |
| SHA-384 | HASH | 2 | quantum-safe | Grover halving leaves 192-bit effective |
| ML-KEM-768 | KEY_ENCAPSULATION | 3 | quantum-safe | NIST FIPS 203 — designed for post-quantum security |
| ML-DSA-65 | DIGITAL_SIGNATURE | 3 | quantum-safe | NIST FIPS 204 — designed for post-quantum security |
| alg:none | UNKNOWN | None | unknown | No signature algorithm — critical vulnerability, not a quantum issue |



### Section 3: Citing the CBOM as Compliance Evidence (audience: auditors, compliance teams)

**NIST SP 800-208 alignment:**

NIST SP 800-208 requires a cryptographic inventory as a precondition for NIST PQC migration planning. The QU.I.R.K. CBOM satisfies this requirement. Suggested audit language:

> *"A cryptographic inventory was produced using QU.I.R.K. v[version] on [date]. The inventory covers [N] endpoints across [network ranges]. Cryptographic algorithms were classified against the NIST Post-Quantum Cryptography evaluation criteria (NIST IR 8413). The complete inventory is attached as CBOM artifact `cbom-[timestamp].cdx.json` in CycloneDX 1.6 format. Quantum-vulnerable components have been identified and are addressed in the accompanying migration roadmap."*

**CNSA 2.0 alignment:**

CNSA 2.0 (NSA, August 2022) requires NSS operators to complete a cryptographic inventory by 2025 and begin algorithm migration by 2026. The QU.I.R.K. CBOM is suitable as the CNSA 2.0 inventory artifact. Suggested language:

> *"Per CNSA 2.0 requirements, a full cryptographic inventory was conducted on [date] using QU.I.R.K. v[version]. The inventory identified [N] quantum-vulnerable components across [N] systems. Specific components mapped to CNSA 2.0 algorithm classes are documented in the attached CBOM. A phased migration plan targeting CNSA 2.0 compliance by [year] is attached separately."*

**ISO 27001 / ISO 27002:2022 alignment:**

ISO 27002:2022 Control 8.24 (Use of cryptography) requires organizations to define rules on the use of cryptographic controls and maintain an inventory of cryptographic assets. The QU.I.R.K. CBOM satisfies the inventory component of Control 8.24. Suggested language:

> *"Consistent with ISO 27002:2022 Control 8.24, a cryptographic asset inventory has been maintained. The current inventory, produced on [date], is available as CBOM artifact `cbom-[timestamp].cdx.json`."*
