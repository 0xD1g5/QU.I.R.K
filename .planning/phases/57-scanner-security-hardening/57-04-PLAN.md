<threat_model>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| operator target list → SAML scanner | Targets may originate from third-party inventories and could include internal/metadata IPs that exfiltrate cloud creds via SSRF. |
| network → metadata service | EC2/GCP/Azure metadata IPs return cloud-instance credentials when fetched from inside the VPC. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-57-10 | Information Disclosure (SSRF) | _fetch_metadata httpx.get | mitigate | Route every URL through validate_external_url before httpx.get. Default rejects RFC1918, link-local, loopback, file://, and metadata IPs. Closes CR-04. |
| T-57-11 | Information Disclosure (cloud cred theft) | metadata-IP fetches | mitigate | 169.254.169.254 and fd00:ec2::254 are blocked unconditionally — even allow_internal=True does not unlock them. Test test_fetch_metadata_allow_internal_still_blocks_metadata_ip asserts this. |
| T-57-12 | Repudiation (silent SSRF) | opt-in internal target fetches | mitigate | When allow_internal_targets=True, the scanner emits a HIGH advisory CryptoEndpoint per affected target naming the URL fetched. The consulting deliverable surfaces exactly which internal hosts were probed. |

**Residual risks:**
- DNS rebinding: hostname targets resolve at fetch time. A hostname that
  resolves to a public IP at validation and an internal IP at fetch could
  bypass the guard. Out of scope for v4.8 (would require DNS-pin + connect-time
  verification).
- HTTP 30x redirects to internal targets: validate_external_url checks the
  initial URL only; httpx.follow_redirects=True may chase to internal hosts.
  Mitigation candidate for v4.9: pass a custom `httpx.Transport` that
  re-validates each redirect Location.

</threat_model>
