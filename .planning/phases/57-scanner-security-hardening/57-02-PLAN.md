<threat_model>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| operator config (YAML/CLI) → scanner runtime | Operator-controlled config dictates whether SSRF / TLS / credential safeguards are active. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-57-04 | Tampering | SecurityCfg defaults | mitigate | All three knobs default False — operator must explicitly opt in. CLI flags can only flip False -> True (opt-in only). YAML `allow_insecure_jwks: true` is preserved across CLI runs (operator can't accidentally re-enable safety just by omitting a flag). |
| T-57-05 | Information Disclosure | broker_credentials YAML | mitigate | Passwords sourced from environment variables via `pass_env` field — never inline in YAML, never echoed to logs. Default empty map means no credentials sent unless operator configures specific host:port. |
| T-57-06 | Repudiation | scan_error_category enum | accept | `invalid_input` rows write the reason code + 32-char redacted preview (D-08); operator-readable but not exfiltration-friendly. Audit trail preserved without leaking attacker payloads. |

**Residual risks:**
- An operator who edits `config_template.yaml` directly and sets all three
  knobs True effectively disables Phase 57's safety net. Acceptable: this is
  the documented opt-in path, and the HIGH advisory findings emitted by Wave 2
  scanners surface the per-target impact.
- `pass_env` env-var values can still leak via process inspection or core
  dumps. Out of scope for v4.8; addressed in part by Phase 59
  (LEAK-01..LEAK-03) credential scrubbing.

</threat_model>
