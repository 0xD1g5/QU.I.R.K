# Phase 999.84 Plan 01: ldaps macOS Host-Mount Compat Summary

Switched the chaos lab's `ldaps` service from `osixia/openldap:1.5.0` to
`bitnamilegacy/openldap:2.6.10-debian-12-r4` to eliminate the macOS Docker
Desktop chown-on-readonly-bindmount restart loop, after first verifying that
the lighter-touch `KEEP_EXISTING_CONFIG=true` env var does not help.

## Branch decision: option (a) fallback fired

The plan ran option (c) → option (a) cascade per CONTEXT D-Bug1.

**Option (c) attempted first** — added `KEEP_EXISTING_CONFIG: "true"` to the
existing `ldaps` service `environment:` block.

Verification on a fresh container produced:

```
$ docker ps --filter "name=chaoslab-ldaps-1" --format '{{.Status}}'
Restarting (1) 14 seconds ago
$ docker inspect chaoslab-ldaps-1 --format '{{.RestartCount}}'
10
$ docker logs chaoslab-ldaps-1 2>&1 | grep -ci "Read-only file system"
700
```

The chown step in osixia's entrypoint runs unconditionally before the
`KEEP_EXISTING_CONFIG` flag is honored — the env var only short-circuits
*later* config-bootstrap work, not the initial recursive chown of
`/container/service/slapd/assets/certs/`. Option (c) was reverted via
`git checkout docker-compose.yml`, leaving zero residual artifact.

**Option (a) executed** — three-file atomic edit per CLAUDE.md chaos-lab
rule:

1. `docker-compose.yml` — image swap + env var remap to bitnami's namespace
   (`LDAP_ROOT`, `LDAP_TLS_CERT_FILE`/`KEY_FILE`/`CA_FILE` with absolute
   paths under `/opt/bitnami/openldap/certs/`, explicit
   `LDAP_LDAPS_PORT_NUMBER: 636`). Service name, profile membership, port
   mapping, and bind-mount source `./certs` all unchanged.
2. `README.md` row 34 — Phase column annotated
   `v4.1; bitnamilegacy/openldap:2.6.10-debian-12-r4 (macOS-compat 2026-05-15)`.
3. `expected_results_v4.md` — `## Profile: ldaps` section gained a
   `> **Note:**` block documenting the image change, why bitnamilegacy/* is
   used, and that crypto findings are unchanged.

image:

```
$ openssl s_client -connect localhost:636 -showcerts < /dev/null 2>&1 \
    | grep -iE "subject=|verify return code"
subject=C=US, ST=NY, L=Lab, O=ChaosLab, OU=Server, CN=modern.chaos.local
Verify return code: 19 (self-signed certificate in certificate chain)
```

Verify code 19 is expected for the lab's self-signed CA — matches the
oracle's `CERT_SELFSIGNED` finding row.

```
$ docker run --rm --network chaoslab_default -e LDAPTLS_REQCERT=never alpine \
    sh -c 'apk add --no-cache openldap-clients >/dev/null && \
           ldapsearch -H ldaps://chaoslab-ldaps-1:636 -x \
                      -b "dc=chaos,dc=local" -LLL "(objectClass=*)"'
dn: dc=chaos,dc=local
objectClass: dcObject
objectClass: organization
dc: chaos
o: example

dn: ou=users,dc=chaos,dc=local
...
dn: cn=user01,ou=users,dc=chaos,dc=local
...
```

`LDAPTLS_REQCERT=never` is required because the cert CN is
`modern.chaos.local` (the lab's shared cert), not the container hostname
`chaoslab-ldaps-1`. This is identical behavior to the previous osixia
image and matches the oracle's documented self-signed-cert posture.

## Deviations from Plan

- **Found during:** Task 4 (initial bring-up of fallback image)
- **Issue:** `docker pull bitnami/openldap:2.6` failed with `not found`.
  Bitnami migrated their free Docker Hub images to a paid Secure Images
  model in late 2025; `bitnami/*` now hosts only paid-tier content.

  `bitnamilegacy/*` namespace is Bitnami's officially-supported free-tier
  successor for the legacy free images. Pinned to a specific tag (not the
  floating `2.6`) for reproducibility.
- **Files modified:** docker-compose.yml, README.md, expected_results_v4.md
  (all three caught the rename in the same atomic commit)
