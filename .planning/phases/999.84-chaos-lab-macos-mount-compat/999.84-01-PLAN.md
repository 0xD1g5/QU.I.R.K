<objective>
Fix Bug 1 (DEF-999.83-A): `chaoslab-ldaps-1` enters a restart loop on macOS Docker Desktop because `osixia/openldap:1.5.0`'s entrypoint chowns bind-mounted cert files at `/container/service/slapd/assets/certs/...`, and macOS exposes those files read-only.

Per CONTEXT D-Bug1, this plan takes a two-step approach with a hard branch decision at Task 2:

1. **Primary fix (option (c)):** Add `KEEP_EXISTING_CONFIG: "true"` to the existing ldaps `environment:` block. Single-line YAML change. `osixia/openldap` honors this to skip its config-bootstrap (and the chown step that goes with it) on subsequent starts. Cost to test: ~5 minutes.

2. **Fallback (option (a)):** If the chown happens BEFORE `KEEP_EXISTING_CONFIG` is read (i.e. the env var is honored too late in the entrypoint to help), switch the image to `bitnami/openldap:2.6` which does not chown bind-mounts on startup. Requires remapping env var names (e.g. `LDAP_ADMIN_PASSWORD` → `LDAP_ADMIN_PASSWORD`, `LDAP_TLS_CRT_FILENAME` → mount path differences) and updating README row 34 + oracle ## Profile: ldaps section in the same commit (CLAUDE.md chaos-lab rule).

The branch decision is made by the executor at Task 2 based on the verification log evidence, not pre-decided here.

Purpose: Restore `ldaps` profile to healthy state on macOS without losing LDAPS/cert behavior; pick the lightest-touch fix that actually works.
Output: One atomic commit. Either `fix(chaos-lab): ldaps macOS-host compat via KEEP_EXISTING_CONFIG (BACK-91 bug 1)` (option c) OR `fix(chaos-lab): switch ldaps to bitnami/openldap for macOS-host compat (BACK-91 bug 1)` (option a). Never both.
</objective>
