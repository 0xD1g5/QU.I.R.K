<objective>
Fix Bug 3 (DEF-999.83-C): `source/seed.sh` is not idempotent. On any `./lab.sh all` invocation where the `chaoslab_gitea_data` named volume persists (which is the normal case once you've done one full run), the first `curl -sf POST /api/v1/user/repos` returns HTTP 409, `curl` exits 22, `set -e` kills the script, and `chaoslab-gitea-seed-1` shows `Exited (22)` instead of `Exited (0)`. The repos and files actually exist correctly — the seed effectively succeeded — but the script failure pollutes the lab UAT signal.

Per CONTEXT D-Bug3 (mixed strategy):
- **Repo creation (3 sites):** use existence check (option (b)) — `GET /api/v1/repos/{user}/{repo}` first, only POST if the response is 404. This preserves real-error visibility: a 5xx response, network failure, or auth failure on the GET will still abort the script via `set -e`, which is what we want.
- **Per-file `put_file` calls (many sites):** append `|| true` to the curl call inside the `put_file` helper (option (a)). For per-file POSTs, a 409 truly means "same path already exists with the same committed content" — the seed is fully deterministic in file content + commit messages — so swallowing the error is safe here. Per-file existence checks would double the API call count for no real benefit.

This mixed approach is what CONTEXT D-Bug3 explicitly recommended; the rationale is captured here so the SUMMARY doesn't have to re-derive it.

The phase 999.83 Plan 01 work (admin user → `labadmin` via env vars + a `gitea-init` one-shot) is preserved as-is. This plan only adds the idempotency wrapping; admin credentials and the readiness loop are untouched.

Purpose: Make seed.sh safe to re-run without volume wipes, so the steady-state lab UAT shows `Exited (0)` for gitea-seed instead of `Exited (22)`.
Output: One atomic commit `fix(chaos-lab): make gitea-seed idempotent on re-run (BACK-91 bug 3)`.
</objective>
