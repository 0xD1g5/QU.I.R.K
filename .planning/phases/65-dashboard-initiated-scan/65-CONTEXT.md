# Phase 65: Dashboard-Initiated Scan - Context

**Gathered:** 2026-05-13 (updated from 2026-05-10)
**Status:** Ready for planning

<domain>
## Phase Boundary

Phase 65 lets an operator configure, launch, and watch a scan to completion entirely from the dashboard â€” closing the "CLI required" primetime gap (BACK-86 slice 1). Three concrete deliverables:

1. **`/scan/new` form** â€” target spec, profile, calibration, nmap toggle; validated against the same Pydantic schema the CLI uses; protected by Phase 58 bearer auth + CSRF.
2. **Backend job dispatch + `/api/jobs` routes** â€” POST creates a `scan_jobs` row and spawns the scan as a subprocess; GET polls job status and current stage; DELETE cancels (SIGTERM).
3. **Live status page `/scan/job/:jobId`** â€” streams scanner stage transitions (7 stages) via client-side polling; auto-navigates to results on completion; shows "Cancel" button.

**In scope:** `scan_jobs` SQLite table, `--job-id` flag in `run_scan.py`, `/api/jobs` router, `/scan/new` page, `/scan/job/:jobId` page, `useJobStatus` hook, post-completion navigation via `setSelectedScanId`, FastAPI lifespan startup hook for stale job recovery.

**Out of scope:** Per-scanner granular toggles (Phase 65 keeps the form simple), scan resumption (Phase 67), scan history / clone (Phase 66), multi-user job ownership, real-time WebSocket streaming (polling is sufficient), operator notification on completion.

</domain>

<decisions>
## Implementation Decisions

### D-01: Scan Job Dispatch Mechanism

Use **`subprocess.Popen`** to invoke `quirk <targets> --profile <profile> --output <dir> --job-id <uuid>` from FastAPI. This is identical in principle to Phase 63 D-02 (scheduler subprocess dispatch):
- Crash-isolated â€” a hung scan does not crash the dashboard API process
- Reuses the full `run_scan.py` code path unchanged (same config, same pipeline, same DB write path)
- PID stored in `scan_jobs.pid` for SIGTERM cancellation

Do NOT use FastAPI `BackgroundTasks` (state sharing risk, no PID for cancellation) or asyncio subprocess (same risk). Pass `--db-path <current_db_path>` so the spawned scan writes to the same SQLite file as all other scans.

### D-02: scan_jobs SQLite Table

New table `scan_jobs` (new SQLAlchemy model in `quirk/models.py`, registered in `quirk/db.py`). Columns:

| Column | Type | Notes |
|---|---|---|
| `job_id` | `String(36)` PK | UUID generated by API on POST |
| `pid` | `Integer` nullable | Set after Popen succeeds; null until started |
| `status` | `String(16)` | `queued â†’ running â†’ completed \| failed \| cancelled` |
| `current_stage` | `String(32)` nullable | E.g. `"discovery"`, `"tls"`, `"ssh"` |
| `target` | `String(512)` | Raw targets string from form |
| `profile` | `String(16)` | `quick \| standard \| deep` |
| `calibration` | `String(16)` | `strict \| balanced \| lenient` |
| `enable_nmap` | `Boolean` | Default False |
| `started_at` | `DateTime` | Set on Popen |
| `completed_at` | `DateTime` nullable | Set when status flips to terminal |
| `scan_run_id` | `String` nullable | The `scan_run_id` from `CryptoEndpoint` rows; filled on completion |
| `error_message` | `Text` nullable | Set on failure |

Registration: add `_ensure_scan_jobs_table(engine)` to `init_db()` in `quirk/db.py` after `_ensure_scheduled_tables(engine)`, using the same `Base.metadata.create_all(engine, checkfirst=True)` pattern.

### D-03: Stage Reporting via --job-id Flag

Add `--job-id <uuid>` as an optional argparse argument to `run_scan.py`. When present, the scan updates `scan_jobs.current_stage` at each major phase transition using a lightweight helper (`quirk/cli/job_progress.py`):

```python
update_job_stage(db_path, job_id, stage)  # opens DB session, updates row, closes
```

Stage names match the `run_scan.py` execution sequence:
1. `discovery`
2. `tls`
3. `ssh`
4. `api`
5. `identity`
6. `data_at_rest`
7. `reports`

On scan completion, `run_scan.py` updates `status = "completed"` and sets `scan_run_id`. On unhandled exception, sets `status = "failed"` and writes `error_message`. The helper is a no-op if `--job-id` is absent (CLI users unaffected).

### D-04: API Routes â€” /api/jobs

New router at `quirk/dashboard/api/routes/jobs.py`:

```
POST   /api/jobs              Create job + spawn subprocess (201; body: ScanSubmitRequest)
GET    /api/jobs/{job_id}     Poll status (200; body: JobStatusResponse)
DELETE /api/jobs/{job_id}     Cancel â€” sends SIGTERM to pid (204 / 404)
```

All routes require Phase 58 bearer auth. `POST` and `DELETE` require CSRF (mutating). `GET` is read-only (no CSRF needed).

`POST /api/jobs` flow:
1. Validate `ScanSubmitRequest`
2. Generate `job_id = str(uuid.uuid4())`
3. Insert `scan_jobs` row with `status="queued"`
4. Call `subprocess.Popen([...quirk args..., "--job-id", job_id])`
5. Update row: `pid = proc.pid`, `status = "running"`, `started_at = now()`
6. Return `201 {"job_id": job_id, "status": "running"}`

`DELETE /api/jobs/{job_id}` flow:
1. Look up `scan_jobs` row; 404 if not found
2. If `pid` is set and status is `running`: `os.kill(pid, signal.SIGTERM)`
3. Update `status = "cancelled"`, `completed_at = now()`
4. Return 204

### D-05: Pydantic Validation Schema

Add `ScanSubmitRequest` to `quirk/dashboard/api/schemas.py`:

```python
class ScanSubmitRequest(BaseModel):
    targets: str = Field(..., min_length=1, max_length=1024)
    profile: Literal["quick", "standard", "deep"] = "standard"
    calibration: Literal["strict", "balanced", "lenient"] = "balanced"
    enable_nmap: bool = False

    @field_validator("targets")
    @classmethod
    def no_file_paths(cls, v: str) -> str:
        if v.strip().startswith("@"):
            raise ValueError("@file paths are not supported from the dashboard â€” use the CLI")
        return v
```

Add `JobStatusResponse` to schemas.py:

```python
class JobStatusResponse(BaseModel):
    job_id: str
    status: str           # queued | running | completed | failed | cancelled
    current_stage: Optional[str]
    started_at: Optional[str]
    completed_at: Optional[str]
    scan_run_id: Optional[str]
    error_message: Optional[str]
    stage_index: int      # 0â€“7, computed from current_stage for progress bar
    stage_total: int      # always 7
```

`targets` validation: accepts comma-separated hosts, CIDRs, single FQDN, IP ranges. The `@field_validator` rejects any value starting with `@` (422 with field-level message) â€” `@file` paths are CLI-only (Phase 58 defense-in-depth). The CLI's `target_expander.py` handles deeper parsing at scan time.

### D-06: Form Fields â€” /scan/new

The form exposes four controls â€” no per-scanner toggles (keeps the form consultant-friendly):

1. **Targets** â€” free-text textarea (single host, comma list, CIDR); placeholder shows host/CIDR/comma examples only â€” no `@file` mention
2. **Profile** â€” radio group: Quick / Standard / Deep (default Standard)
3. **Calibration** â€” radio group: Strict / Balanced / Lenient (default Balanced)
4. **Enable nmap discovery** â€” checkbox (default off)

On submit: validate client-side (non-empty targets), then POST to `/api/jobs`. Show inline error if 422. Navigate to `/scan/job/:jobId` on 201.

### D-07: Client-Side Polling â€” useJobStatus Hook

New hook `src/dashboard/src/hooks/useJobStatus.ts`. Polls `GET /api/jobs/{job_id}` every **3 seconds** following the Phase 62 cancellation-safe pattern:

```typescript
useEffect(() => {
  let cancelled = false
  const poll = async () => {
    if (cancelled) return
    const data = await fetchJobStatus(jobId)
    if (!cancelled) setJobStatus(data)
    if (!cancelled && !isTerminal(data.status)) {
      setTimeout(poll, 3000)
    }
  }
  poll()
  return () => { cancelled = true }
}, [jobId])
```

Stops polling when status is `completed | failed | cancelled` (terminal states).

### D-08: Post-Completion Navigation

When `useJobStatus` returns `status === "completed"` and `scan_run_id` is set:
1. Call `setSelectedScanId(scan_run_id)` (from `useSelectedScan` hook)
2. Call `navigate("/")` (React Router) to show the executive summary for the new scan

The new scan is already in the DB and will appear in `ScanSelector` on the next `useScanList` refresh. The scan is indistinguishable from a CLI-launched scan in storage and reporting (success criterion 3).

### D-09: Cancellation Button

The `/scan/job/:jobId` status page shows a "Cancel scan" button when `status === "running"`. On click: `DELETE /api/jobs/{job_id}`. On 204: update local status to `"cancelled"` and navigate back to `/scan/new` (with a toast/notice).

### D-10: Auth Wiring

All `/api/jobs` routes mount with `dependencies=[Depends(require_auth)]`. The `POST` and `DELETE` routes additionally use `Depends(require_csrf)` â€” same pattern as Phase 63's `router = APIRouter(dependencies=[Depends(require_auth), Depends(require_csrf)])` in `schedules.py`. `GET /api/jobs/{id}` is read-only; it only needs `require_auth`.

### D-11: React Route Registration

Two new React Router routes under `src/dashboard/src/App.tsx` (or equivalent route file):
- `/scan/new` â†’ `<ScanNewPage />`
- `/scan/job/:jobId` â†’ `<ScanJobPage />`

Add "New Scan" as a `Button variant="default"` at the **top of the sidebar**, above all view links â€” visually distinct as the primary CTA, navigates to `/scan/new`. This is an action, not a view link, so it should not be styled identically to Executive/Findings/etc. nav items.

### D-12: FastAPI Lifespan â€” Stale Job Recovery

Introduce a `@asynccontextmanager` lifespan function in `quirk/dashboard/api/app.py`. The startup block runs a stale-job sweep before the app starts serving:

```python
@asynccontextmanager
async def lifespan(app: FastAPI):
    db_path = app.state.db_path
    _recover_stale_jobs(db_path)  # flip runningâ†’failed on restart
    yield
    # (no teardown needed)
```

`create_app(db_path)` sets `app.state.db_path` before constructing the FastAPI instance, then passes `lifespan=lifespan` to the `FastAPI(...)` constructor. `_recover_stale_jobs` queries `scan_jobs WHERE status = 'running'` and flips them to `status = 'failed'`, `error_message = 'API restarted â€” job lost'`, `completed_at = now()`.

### Claude's Discretion

- **Output directory for spawned scans:** `output/jobs/{job_id}/` â€” separates dashboard-initiated output from interactive scan output, same philosophy as Phase 63's `output/scheduled/`.
- **Signal handling:** after SIGTERM, the API marks status `"cancelled"` immediately (optimistic). If the process is already gone (race), `os.kill` raises `ProcessLookupError`; catch it silently.
- **Stage progress bar:** compute `stage_index` from `current_stage` string on the backend (returned in `JobStatusResponse`) so the frontend doesn't need to maintain a stage-to-index mapping.
- **Loading state on `/scan/job/:jobId`:** use `<PageSpinner />` until first poll resolves.
- **`/scan/job/:jobId` with unknown job ID:** if `GET /api/jobs/{job_id}` returns 404, show an error state ("Scan not found") with a link back to `/scan/new` â€” do not redirect silently.

<specifics>
## Specific Ideas

- Success criterion 3 is explicit: the new scan must be **indistinguishable from a CLI-launched scan** in storage and reporting. This means the same `scan_run_id` pattern, same `CryptoEndpoint` rows, same CBOM output. The `--job-id` flag is purely progress-tracking overhead â€” the actual scan output path is unchanged.
- The dashboard currently shows "Dashboard API â€” strictly read-only" in ARCHITECTURE.md. This phase (and Phase 63) change that. Phase 65 is the second writable route family (Phase 63 was first). The architecture doc should be updated as part of this phase.
- Stage display in the UI: show a step indicator (1 of 7) with stage name. Each stage transition is a DB update from the subprocess via `update_job_stage()`. The progress bar fills as stages complete.
- **`@file` security note:** `@file` targets are CLI-only by design. The `ScanSubmitRequest.no_file_paths` validator rejects any target starting with `@` with a clear 422 message. The form placeholder should not mention `@file` at all.
- **Lifespan pattern note:** `app.py` has no startup hook today. Phase 65 introduces the first FastAPI lifespan. Use `@asynccontextmanager` from `contextlib` â€” `from contextlib import asynccontextmanager`. `app.state.db_path` is set in `create_app(db_path)` before the FastAPI instance is returned.

</specifics>
