## Deviations from plan

### [Rule N/A — scope-boundary] Strict UAT criterion not met; three pre-existing macOS-host issues logged as deferred

- **Found during:** Task 2 global UAT.
- **Issue:** Three containers fail the strict "zero `Exited (1)/(2)`, zero `unhealthy`" criterion:
  1. `chaoslab-ldaps-1` in `Restarting (1)` loop — `osixia/openldap:1.5.0` tries to chown bind-mounted cert files; Docker Desktop on macOS exposes them read-only.
  2. `chaoslab-rabbitmq-broker-1` Exit — `Error when reading /var/lib/rabbitmq/.erlang.cookie: eacces`. macOS Docker Desktop bind-mount uid/gid mapping conflict.
  3. `chaoslab-gitea-seed-1` Exit (22) — `curl -sf POST /api/v1/user/repos` returns 22 on HTTP 409 when re-running against a pre-populated Gitea volume. `set -e` kills the script. Repos already exist and are queryable, so the seed effectively succeeded on the first run.
- **Why deferred not auto-fixed:** None of these are caused by the four BACK-90 fixes or Plan 05's docs changes. Per the executor Scope Boundary rule ("Only auto-fix issues DIRECTLY caused by the current task's changes"), they are out-of-scope.
- **Action:** Logged to `.planning/phases/999.83-chaos-lab-service-config-drift/deferred-items.md` as DEF-999.83-A/B/C with suggested follow-ups (switch ldaps image to one that doesn't chown bind-mounts, use named volume + `RABBITMQ_ERLANG_COOKIE` env for rabbitmq, add idempotency guard to `source/seed.sh`).
- **Phase scope verification:** All four BACK-90 fix sites + the deprecated storage profile removal + image-pin policy verify clean. The phase's stated goal is met.
