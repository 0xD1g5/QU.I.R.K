<objective>
Fix Bug 2 (minio-seed KMS failure, profile: storage-s3) by configuring the `minio` service with a KMS secret key so the seed script's `mc encrypt set sse-s3 local/encrypted-bucket` succeeds, and formalize the image pin per CONTEXT's cross-cutting "no floating tags" rule.

Root cause (from v4.8-ROADMAP.md BACK-90 row + confirmed log evidence `Unable to enable auto encryption. Server side encryption specified but KMS is not configured.`): Recent MinIO releases enforce that SSE-S3 requires a configured KMS. The current `minio` service in docker-compose.yml (lines 892-907) sets only `MINIO_ROOT_USER`/`MINIO_ROOT_PASSWORD`. `storage/minio-seed.sh` line 14 calls `mc encrypt set sse-s3 local/encrypted-bucket` — this now fails because MinIO has no KMS backend.

Decision (CONTEXT D-2 option (a) — chosen): Set `MINIO_KMS_SECRET_KEY=lab-key:<base64-encoded-32-bytes>` on the `minio` service. This provides a static built-in KMS that MinIO uses for SSE-S3 — exactly the production-realistic pattern an enterprise would use with a real external KMS. Option (b) (strip auto-encryption from seed.sh) was rejected because it would force changes to `expected_results_v4.md` profile-storage-s3 row 381 ("encrypted-bucket | SSE-S3 (AES256) | S3-managed") which is intentionally there to validate the no-finding path for the AWS connector's SSE-S3 detection.

Also pin the image: replace `minio/minio:latest` with a dated stable tag per CONTEXT cross-cutting locked decision ("no `:latest`, no floating major-version tags").

Purpose: Restore `storage-s3` profile to healthy state without losing the encrypted-bucket scan target that the oracle and AWS connector depend on.
Output: Two surgical edits to one compose service block, one atomic commit `fix(chaos-lab):` prefix.
</objective>
