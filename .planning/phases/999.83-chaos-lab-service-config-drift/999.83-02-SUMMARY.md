# Phase 999.83 Plan 02: MinIO SSE-S3 KMS configuration (BACK-90 bug 2) Summary

Fix Bug 2 of BACK-90: the `minio-seed` container's `mc encrypt set sse-s3 local/encrypted-bucket`
call was failing with `Unable to enable auto encryption. Server side encryption specified but
KMS is not configured.` because recent MinIO releases require an explicit KMS for SSE-S3.
Added `MINIO_KMS_SECRET_KEY=lab-key:<base64-32-bytes>` to the `minio` service environment so
MinIO has a built-in static KMS, and replaced `image: minio/minio:latest` with a dated stable
pin per the cross-cutting "no floating tags" rule.

## Root cause

`docker-compose.yml` lines 925–940 (post-Plan-01 layout) defined the `minio` service with only
`MINIO_ROOT_USER` and `MINIO_ROOT_PASSWORD`. No `MINIO_KMS_*` env vars meant MinIO booted
without a KMS backend. When `minio-seed` ran its line-14 call
`mc encrypt set sse-s3 local/encrypted-bucket`, MinIO rejected the request because SSE-S3
requires a configured KMS to wrap object keys. The seed exited non-zero, and the
`storage-s3` profile never reached a healthy steady state.

## Fix (CONTEXT D-2 option (a))

Two surgical edits to the `minio:` service block in
`quantum-chaos-enterprise-lab/docker-compose.yml`:

1. **Image pin.** `image: minio/minio:latest` → `image: minio/minio:RELEASE.2025-09-07T16-13-09Z`.
   This is the most recent stable, non-`cpuv1` RELEASE tag published to Docker Hub
   (`minio/minio` repo) at execution time, verified via the Docker Hub tags API on 2026-05-15.
2. **KMS env.** Added a single line to the `environment:` block:
   ```yaml
   MINIO_KMS_SECRET_KEY: "lab-key:e6QwR1eRz0t8u3y2z6mP/m4G2D8jH7yB3KvN0aN8z7M="
   ```
   Format is `<key-name>:<base64-32-bytes>`. The base64 payload is stable lab-only entropy —
   explicitly NOT a secret, never used outside the chaos lab — and may be committed.

`storage/minio-seed.sh` was deliberately not modified: the seed's encryption call is correct
once the server has a KMS. The `unencrypted-bucket` HIGH-finding path is unchanged (oracle
row 382 still validates).

### Why not option (b)?

Per CONTEXT D-2, option (b) (strip SSE-S3 setup from `minio-seed.sh`) would force changes to
`expected_results_v4.md` profile-storage-s3 row 381 (`encrypted-bucket | SSE-S3 (AES256) |
S3-managed`), which is the no-finding-path scan target for `aws_connector.py` line 257.
Removing that target would lose meaningful AWS-connector coverage. Option (a) preserves the
target with a production-realistic config pattern.

### RELEASE tag selection

Looked up `https://hub.docker.com/v2/repositories/minio/minio/tags/?page_size=20&name=RELEASE`
on 2026-05-15. Top stable (non-`-cpuv1`) result:

```
RELEASE.2025-09-07T16-13-09Z
RELEASE.2025-07-23T15-54-02Z
RELEASE.2025-07-18T21-56-31Z
RELEASE.2025-06-13T11-33-47Z
...
```

Chose `RELEASE.2025-09-07T16-13-09Z` — the most recent stable tag — to maximize the time
before the pin needs revisiting. Verified the tag exists by pulling it (image came up
healthy in verification).
