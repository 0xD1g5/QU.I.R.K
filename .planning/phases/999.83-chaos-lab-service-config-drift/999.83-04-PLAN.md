<objective>
Fix Bug 4 (mysql-ssl-off `--skip-ssl` rejection, profile: database) by pinning the image to `mysql:8.0`, which preserves the `--skip-ssl` flag the chaos lab depends on. The currently-floating `mysql:8` tag resolves to MySQL 8.4.x (`mysql:8.4.9` per the failure log), where `--skip-ssl` was removed.

Root cause (v4.8-ROADMAP.md BACK-90 row + confirmed log evidence `unknown option '--skip-ssl'`): docker-compose.yml line 875 uses `image: mysql:8`. Docker Hub's `mysql:8` floating tag now points to 8.4.x. MySQL 8.4 removed `--skip-ssl` in favor of `--require-secure-transport=OFF`.

Decision (CONTEXT D-4 recommended option (a) — chosen): **Pin to `mysql:8.0`**. Rationale (3 sentences):
1. The chaos lab's purpose is to expose the insecure-by-default behavior consultants actually find in the wild; MySQL 8.0 is still what most enterprises run in 2026, and `--skip-ssl` on 8.0 is the documented insecure config a real audit would surface.
2. Option (b) — pin to 8.4 with `--require-secure-transport=OFF` — means tracking yet another moving target as MySQL evolves its TLS-disable knobs; option (a) freezes the lab against a stable, widely-deployed image.
3. Either option satisfies the ROADMAP success criterion (scanner must continue to detect the service as SSL-off / plaintext); option (a) requires the minimum diff.

Also formalizes the image pin per CONTEXT Cross-cutting locked decision ("no `:latest`, no floating major-version tags") — replacing `mysql:8` (floating, slid 8.0→8.4) with the explicit `mysql:8.0` minor-major pin.

Purpose: Restore `database` profile to healthy state without rewriting the scanner detection logic or changing the oracle.
Output: Single-line compose edit, one atomic commit `fix(chaos-lab):` prefix.
</objective>
