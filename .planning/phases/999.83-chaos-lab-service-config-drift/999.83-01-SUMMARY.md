# Phase 999.83 Plan 01: Gitea root-user crash (BACK-90 bug 1) Summary

Fix Bug 1 of BACK-90 (chaos lab service config drift): the `gitea` service in
`quantum-chaos-enterprise-lab/docker-compose.yml` crashed on startup with
`[F] Gitea is not supposed to be run as root` because the custom `command:`
bash wrapper overrode the image entrypoint and never dropped to the `git`
user. Replaced with a one-shot `gitea-init` sidecar that creates the admin
user (`labadmin/admin123`) via `gitea admin user create` running as the `git`
user against the shared `gitea_data` volume after `gitea` is healthy.

## Root cause

`gitea/gitea:1.21+` refuses to run as root since 1.21.x. The original compose
block (docker-compose.yml lines 629–635) wrapped startup with:

```yaml
command: >
  bash -c "
    /usr/local/bin/gitea web &
    sleep 10 &&
    gitea admin user create --admin --username admin --password admin123 ... 2>/dev/null || true &&
    wait
  "
```

That `command:` field overrode the image's `/usr/bin/entrypoint`, which is
the script responsible for chowning `/data` and `exec su-exec git` to drop
privileges. With the entrypoint bypassed, gitea launched as PID 0 / uid 0
and tripped the safety check.

A second, latent bug surfaced once the root issue was fixed: Gitea 1.21
reserves the username `admin` (`Command error: CreateUser: name is reserved`).
The original wrapper masked this with `2>/dev/null || true`, so the lab had
never actually been able to seed repos against `admin` — `gitea-seed` was
silently failing for as long as the image had been at 1.21.x.

## Fix (chosen mechanism — option (c) variant)

CONTEXT D-1 locked three constraints: (1) no downgrade from `gitea/gitea:1.21`,
(2) no docker-socket mount, (3) admin creation moves out of the gitea
`command:` block. The planner recommended option (c) (set
`GITEA__service__DEFAULT_ADMIN_USERNAME` / `_PASSWORD` / `_EMAIL` on the
gitea service so the entrypoint auto-provisions on first start).

Empirically, the Gitea 1.21 image entrypoint does **not** read those
`DEFAULT_ADMIN_*` env vars for auto-provisioning — they only seed `app.ini`
`[service]` defaults used during registration. Tested twice on a freshly
wiped `chaoslab_gitea_data` volume; gitea came up healthy but
`/api/v1/user` returned `user does not exist [uid: 0, name: admin]`.

Fell back to a structural variant of option (c) that preserves all three
CONTEXT D-1 locks:

1. Added a `gitea-init` one-shot container (image `gitea/gitea:1.21`, shared
   `gitea_data` volume, `restart: "no"`).
2. `gitea-init.depends_on: gitea: service_healthy` — runs after gitea web has
   bootstrapped `app.ini` + sqlite schema.
3. Init's `command` uses `su git -c "gitea admin user create ..."` so the CLI
   runs as `git` (satisfies the root-user safety check), is idempotent
   (`gitea admin user list | grep -qE labadmin`), and creates `labadmin/admin123`.
4. `gitea-seed.depends_on` chained to `gitea-init: service_completed_successfully`.
5. `source/seed.sh` `ADMIN_USER` changed to `labadmin`; `expected_results_v4.md`
   clone command updated `admin/crypto-antipatterns-python` -> `labadmin/...`.

Image pin stays at `gitea/gitea:1.21` (CONTEXT D-1 locked, no downgrade).
No docker socket mount. No API bootstrap dance.

## Deviations from plan

**[Rule 4 -> in-scope variant] Switched mechanism (c) to init-sidecar.**
- **Found during:** Task 2 verification.
- **Issue:** The plan's locked mechanism (`GITEA__service__DEFAULT_ADMIN_*`
  env vars on the gitea service) does not work in `gitea/gitea:1.21`. Two
  fresh-volume runs both produced a healthy gitea but no admin user.
- **Fix:** Removed the three `DEFAULT_ADMIN_*` env vars (they were no-ops),
  added a `gitea-init` one-shot container that creates the admin via
  `gitea admin user create` as the `git` user after `gitea` is healthy.
  Stays within CONTEXT D-1's three locks (image pin kept, no socket mount,
  admin creation outside the gitea `command:` block).
- **Files modified:** `quantum-chaos-enterprise-lab/docker-compose.yml`.

**[Rule 1 - Bug] Renamed admin user `admin` -> `labadmin`.**
- **Found during:** Task 2 verification (`Command error: CreateUser: name is reserved`).
- **Issue:** Gitea 1.21 reserves the username `admin`. The original lab's
  `gitea admin user create --username admin` had been silently failing since
  the image moved to 1.21.x (the old `2>/dev/null || true` swallowed it).
- **Fix:** Provisioned `labadmin` instead. Updated `source/seed.sh` and
  the `expected_results_v4.md` clone command to match. Password (`admin123`)
  and email (`admin@chaos.local`) unchanged.
- **Files modified:** `quantum-chaos-enterprise-lab/source/seed.sh`,
  `quantum-chaos-enterprise-lab/expected_results_v4.md`.

**[CLAUDE.md chaos-lab maintenance] Oracle update.**
- `expected_results_v4.md` line 220 had the only repo-clone reference that
  embedded the admin username; updated `admin/crypto-antipatterns-python`
  -> `labadmin/crypto-antipatterns-python` to keep the oracle in sync with
  the new compose state. README.md required no changes (port `20006:3000`
  and service membership unchanged).
