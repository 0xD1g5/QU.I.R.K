<objective>
Fix Bug 1 (Gitea root-user crash, profile: source) by removing the imperative `command:` wrapper that bypasses the image entrypoint's user-drop logic and provisioning the admin user declaratively via Gitea's first-start env vars.

Root cause (from v4.8-ROADMAP.md BACK-90 row + confirmed log evidence `[F] Gitea is not supposed to be run as root`): `gitea/gitea:1.21+` refuses to run as root since 1.21.x. The current compose `command: bash -c "gitea web && gitea admin user create ..."` (docker-compose.yml lines 629-635) overrides the image entrypoint and never drops to the `git` user, tripping the security check.

Decision (CONTEXT D-1 option (c)): Set `GITEA__service__DEFAULT_ADMIN_USERNAME=admin`, `GITEA__service__DEFAULT_ADMIN_PASSWORD=admin123`, `GITEA__service__DEFAULT_ADMIN_EMAIL=admin@chaos.local` on the gitea service. Gitea's image entrypoint provisions the admin on first start, drops to `git`, and the existing `source/seed.sh` (which already uses `admin/admin123` Basic auth) keeps working unchanged.

Purpose: Restore `source` profile to healthy state without downgrading gitea, without mounting the docker socket, and without an API-bootstrap dance.
Output: Surgical compose edit (~10 lines), unchanged seed.sh (verified compatible), one atomic commit `fix(chaos-lab):` per repo convention.
</objective>
