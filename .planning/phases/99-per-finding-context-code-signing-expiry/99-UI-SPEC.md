# Phase 99 — UI Design Contract

> Visual and interaction contract for Phase 99: Per-Finding Context + Code-Signing Expiry.
> Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Scope Clarification

Phase 99 is a **backend enrichment + rendering** phase. The UI surface delta is:

1. **HTML report template** (`report.html.j2`) — add a `Quantum Risk` column to the "All Findings" table in the Technical Appendix; add the same field as an inline block below Description in the "Top Findings" summary table. No new components; additions stay within the existing embedded `<style>` block.
2. **CLI markdown** (`technical.py`) — add a `Quantum Risk` column to the findings table (after Description, before Recommendation). Existing pipe-table format extended; no new terminal primitives.
3. **PDF** — derived from HTML via Playwright; no separate PDF CSS work (Phase 100 owns PDF branding). The HTML additions render to PDF automatically.
4. **React dashboard** — NO layout changes. Dashboard does not render the per-finding table; no new components.

All design decisions below apply to the HTML report template and CLI markdown surfaces unless otherwise noted.

## Copywriting Contract

### Per-Finding Quantum Risk "So What" (CTX-01) — ALGO_IMPACT_MAP Extension

One quantum_risk sentence per crypto class, to be stored in an extended `ALGO_IMPACT_MAP` tuple (adding a third element: `quantum_risk_sentence`). These are the locked strings; executor must use verbatim.

| Crypto Class        | quantum_risk sentence                                                                                                        |
|---------------------|------------------------------------------------------------------------------------------------------------------------------|
| RSA                 | "RSA key material is vulnerable to Shor's algorithm — a sufficiently powerful quantum computer can factor the modulus and recover the private key, breaking both confidentiality and non-repudiation." |
| ECC / ECDSA         | "Elliptic-curve key material is vulnerable to Shor's algorithm — quantum computers can solve the discrete logarithm problem and recover the private key, compromising authentication and forward secrecy." |
| DH / DHE            | "Diffie–Hellman key exchange is vulnerable to Shor's algorithm — a quantum adversary can solve the discrete log problem on recorded sessions, retroactively decrypting captured traffic." |
| DSA                 | "DSA signatures rely on the discrete logarithm problem, which Shor's algorithm breaks — a quantum attacker can forge signatures and impersonate the signing entity." |
| WEAK_HASH / MD5     | "MD5 collision resistance is already broken classically; quantum speedups (Grover's algorithm) further halve the effective security margin, making collision attacks feasible with modest resources." |
| SHA1 / SHA-1        | "SHA-1 collision resistance is broken classically; Grover's algorithm halves the quantum bit-security to ~40 bits, making pre-image and collision attacks practical for state-level adversaries." |
| WEAK_KEY_EXCHANGE / DHE_EXPORT | "Export-grade and short-parameter key exchange can be broken in real time by a classical adversary today; quantum acceleration makes recovery of session keys trivially fast." |
| RC4                 | "RC4 stream cipher is cryptographically broken classically; even without quantum acceleration it provides no meaningful confidentiality — remediation is overdue." |
| 3DES / DES          | "Triple-DES and DES use 56–168-bit key lengths that Grover's algorithm reduces to at most 84 effective bits — insufficient for any security boundary beyond 2030." |
| CODESIGN_EXPIRY     | "An expired code-signing certificate breaks the trust chain — any software signed by this certificate can no longer be verified as authentic, enabling supply-chain attacks or silent malware delivery." |
| CODESIGN_APPROACHING_EXPIRY | "A code-signing certificate expiring within 90 days creates operational risk — if not renewed before expiry, software signed by this certificate will fail verification, blocking deployments and breaking trust." |

**Default fallback (no crypto-class match):**
"This cryptographic weakness reduces the security margin against quantum-capable adversaries. Migrate to NIST-approved post-quantum algorithms per NIST IR 8547."

### Per-Finding Remediation Catalog (CTX-02) — Centralized Catalog Entries

Weakness-specific remediation strings keyed by crypto class. Executor must populate the remediation catalog with these verbatim strings.

| Crypto Class        | Remediation (specific, not generic boilerplate)                                                                             |
|---------------------|-----------------------------------------------------------------------------------------------------------------------------|
| RSA                 | "Replace RSA keys with NIST PQC standard algorithms: ML-KEM (FIPS 203) for key encapsulation or ML-DSA (FIPS 204) for digital signatures. Prioritize certificates and TLS endpoints first." |
| ECC / ECDSA         | "Replace ECDSA/ECDH keys with ML-DSA (FIPS 204) for signatures or ML-KEM (FIPS 203) for key encapsulation. During transition, deploy hybrid TLS (X25519+ML-KEM) as an intermediate step." |
| DH / DHE            | "Disable finite-field Diffie–Hellman key exchange. Configure TLS to prefer X25519 (classical) or X25519+ML-KEM hybrid groups; disable DHE cipher suites in the server's TLS configuration." |
| DSA                 | "Disable DSA. Replace DSA-based SSH host keys and certificates with Ed25519 (short-term) or ML-DSA (FIPS 204) when library support is available." |
| WEAK_HASH / MD5     | "Replace MD5 in all signing, integrity, and authentication contexts. Use SHA-256 or SHA-3-256 as minimum. Reject MD5-signed certificates at the trust-store level." |
| SHA1 / SHA-1        | "Replace SHA-1 signatures with SHA-256 or stronger. Reissue any SHA-1-signed certificates. Configure TLS to reject SHA-1 in the signature_algorithms extension." |
| WEAK_KEY_EXCHANGE / DHE_EXPORT | "Remove all export-grade cipher suites from TLS configuration. Disable DHE_EXPORT and similar suites. Apply OS and application TLS hardening guides (e.g., Mozilla SSL Config Generator — Intermediate or Modern profile)." |
| RC4                 | "Disable RC4 in all TLS and non-TLS contexts. RC4 is prohibited by RFC 7465. Replace with AES-128-GCM or AES-256-GCM cipher suites." |
| 3DES / DES          | "Disable 3DES and DES cipher suites (Sweet32 / NIST SP 800-131A Rev 2 disallowed after 2023). Replace with AES-GCM. Verify no legacy application dependency remains before removal." |
| CODESIGN_EXPIRY (expired) | "Renew the expired code-signing certificate immediately and re-sign all artifacts. Revoke the expired certificate via the issuing CA and update any pinned certificate references in build pipelines." |
| CODESIGN_APPROACHING_EXPIRY | "Renew this code-signing certificate before the not_after date. Update automated renewal policy and alert thresholds to trigger at 90 days remaining to prevent future expiry." |

**NIST_IR_8547_DEPRECATION boilerplate review (D-05):** The existing auto-appended sentence ("Per NIST IR 8547, RSA and ECC are deprecated after 2030 and disallowed after 2035.") must be removed from the generic `quantum_vulnerable=True` append path in `_build_finding` for any finding that has a catalog-sourced remediation. The catalog entries above incorporate the relevant NIST reference inline where appropriate. Only findings without a catalog match should retain the boilerplate suffix.

## Interaction Contract

### HTML Report (static document — no JS interactions)

All Phase 99 additions are static HTML/CSS. No JavaScript or interactive elements added.

**"All Findings" table (Technical Appendix)** — Add a `Quantum Risk` column as the 7th column, after Recommendation. Column renders the `quantum_risk` field value in `--text-muted`, prefixed by the `.quantum-risk-label` inline tag ("Quantum Risk:") in `--accent`. If `quantum_risk` is absent or empty, the cell renders the fallback copy (see default fallback in Copywriting Contract above).

**"Top Findings" table (Executive Summary)** — Do NOT add a 5th column (the summary table is already narrow). Instead, render `quantum_risk` as a `.quantum-risk-block` sub-element within the Description `<td>`, positioned below the truncated description text. Visually separated by a 4px top gap.

**Code-signing expiry findings** — Rendered in the existing finding tables using the standard row structure. No special layout treatment beyond severity badge color (HIGH → `--high`, MEDIUM → `--medium`).

### CLI Markdown Findings Table

Extend the existing pipe-table format in `technical.py`:

```
| Severity | Host | Port | Title | Description | Quantum Risk | Recommendation |
```

`Quantum Risk` column placed between Description and Recommendation. If `quantum_risk` is absent, render the fallback string truncated to 120 characters (matching existing Description truncation pattern).

### React Dashboard (no interaction changes)

No new interactive elements. Dashboard does not render the per-finding table; Phase 99 has no dashboard surface area.
