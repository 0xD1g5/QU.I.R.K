#### 1. ROADMAP SC-2 Disposition: Transaction Rollback vs. Idempotency

**Test:** Review the ROADMAP success criterion SC-2 and decide whether idempotency satisfies the stated requirement.

ROADMAP SC-2 says: "every ALTER TABLE in init_db() executes inside a transaction; a failed migration rolls back cleanly with no partial column additions."

The implementation delivers: idempotency (each _ensure_* function checks `if col not in existing` before ALTER TABLE; re-running init_db() on a partial DB catches up to the full schema). No outer transaction wrapper was added.

CONTEXT.md D-05 explicitly documents this substitution: "Do not add a redundant outer transaction (SQLite nested transactions are complex); instead, make each individual _ensure_* function idempotent and add an integration test."

**Expected:** Either (a) confirm that idempotency satisfies the ROADMAP SC-2 intent and mark SC-2 passed, or (b) note that SC-2 literal rollback guarantee is not met and plan a follow-up fix.

**Why human:** This is an architectural tradeoff decision documented in CONTEXT.md. The tests verify idempotency. Whether "idempotency = rollback" for this codebase's operational requirements is a product decision, not a code-verifiable fact.

#### 2. REVIEW CR-01: strftime Millisecond vs. Microsecond Precision Gap

**Test:** Evaluate the severity of the strftime precision mismatch documented in 64.1-REVIEW.md CR-01.

The issue: SQLite `strftime('%f')` returns 3-digit millisecond strings. `_fetch_session_endpoints` uses a `timedelta(microseconds=1)` window. For a scan with `scanned_at=datetime(2026,5,10,12,0,0,100500)` (100500 microseconds):
- strftime returns `'2026-05-10 12:00:00.101'` (rounds up)
- fromisoformat gives `microsecond=101000`
- The 1-microsecond window `[101000, 101001)` does NOT contain the original 100500
- _fetch_session_endpoints returns 0 results; get_trends silently skips the session

The tests pass because they use `TS_A = datetime(..., 100000)` and `TS_B = datetime(..., 200000)` — exact millisecond multiples that don't trigger the rounding.

**Expected:** Either (a) confirm the practical risk is acceptable (datetime.utcnow() produces millisecond-aligned values in practice) and defer REVIEW CR-01 to Phase 65, or (b) require REVIEW CR-01 fix before Phase 65 begins.

**Why human:** This requires knowledge of whether real scan timestamps in production are ever non-round-millisecond, and whether the silent-skip failure mode is acceptable given Phase 64's trend feature is currently in UAT.
