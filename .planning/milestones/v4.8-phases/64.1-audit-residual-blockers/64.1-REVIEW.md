## Summary

This phase delivered targeted fixes for five audit blockers: `_extract_algo_from_rule_id`
DES/3DES ordering (CR-03), `status_report` date-arithmetic comparison (BL-03), years-input
clamp at both interactive entry points (BL-04), sub-second session disambiguation in the
trends pipeline (CR-05), and `init_db()` idempotency (CR-08). The source-code fixes are
generally correct. The primary concern is a factual mismatch between the documented precision
guarantee ("microsecond") and the actual precision delivered by SQLite's `strftime` function
(millisecond). A secondary concern is that the years-clamp behavioral tests exercise inlined
copies of the logic rather than the actual functions, providing weaker regression coverage than
they appear to. Several minor quality issues are also noted.

## Critical Issues

### CR-01: `strftime('%Y-%m-%d %H:%M:%f')` delivers millisecond precision, not microsecond — fetch window mismatch possible

**File:** `quirk/dashboard/api/routes/trends.py:49`, `quirk/intelligence/trends.py:94`

**Issue:** SQLite's `strftime('%f')` format specifier produces exactly **3 decimal digits**
(milliseconds), not 6 (microseconds). The module docstrings, inline comments, and test comment
(line 70 of `test_trends_subsecond_sessions.py`) all assert "microsecond precision", which is
incorrect.

The practical consequence is a data-consistency hole between `_list_session_timestamps` (which
groups by the 3-digit truncated/rounded string) and `_fetch_session_endpoints` (which uses a
1-microsecond window starting from the `fromisoformat`-recovered timestamp):

1. `scanned_at = datetime(2026, 5, 10, 12, 0, 0, 100500)` is stored by SQLAlchemy as
   `'2026-05-10 12:00:00.100500'` (6 digits).
2. `strftime('%Y-%m-%d %H:%M:%f', scanned_at)` returns `'2026-05-10 12:00:00.101'` (rounds
   up — verified in SQLite).
3. `fromisoformat('2026-05-10 12:00:00.101')` returns `datetime(..., microsecond=101000)`.
4. The 1-microsecond window `[101000, 101001)` does **not** contain `100500` — so
   `_fetch_session_endpoints` returns **zero endpoints** for a session whose timestamp exists
   in the group-by result.
5. `get_trends_timeline` silently skips sessions with zero endpoints (`if not eps: continue`).
6. `get_trends` may report zero or one session when two exist.

The test uses `TS_A = 100000` and `TS_B = 200000` (exact millisecond boundaries), which do
not trigger this rounding: `strftime` returns `.100` / `.200` exactly, and the `fromisoformat`
round-trip gives exact `100000` / `200000` microsecond values. The tests pass but do not
exercise the problematic sub-millisecond-boundary case.

**Fix:** Either (a) align the fetch window to millisecond granularity by rounding `target_ts`
to the nearest millisecond and using a 1-millisecond window, or (b) use a direct string-based
equality filter in `_fetch_session_endpoints` that uses the same `strftime` format as the
group-by:

```python
# Option (b) — string equality, consistent with the group-by key
ts_str = target_ts.strftime("%Y-%m-%d %H:%M:%S.") + f"{target_ts.microsecond // 1000:03d}"
endpoints = (
    db.query(CryptoEndpoint)
    .filter(
        func.strftime("%Y-%m-%d %H:%M:%f", CryptoEndpoint.scanned_at) == ts_str,
        CryptoEndpoint.scanned_at.isnot(None),
    )
    .all()
)
```

Update the test to add a sub-millisecond boundary case (e.g., `100500us` vs `100800us`) and
correct the comment at line 70 of `test_trends_subsecond_sessions.py`.

## Warnings

### WR-01: Years-clamp behavioral tests exercise inlined copies, not the actual source functions

**File:** `tests/test_operator_context_years_clamp.py:21-59`

**Issue:** `_parse_years_interactive` and `_parse_years_operator_context` are hand-written
replicas of the years-parsing block, not calls to the real code paths. The `TestYearsClamp`
parametrize suite (`test_empty_input_defaults_to_7`, `test_zero_raises`, etc.) tests these
inline helpers, not `quirk.interactive.interactive_config` or
`quirk.assessment.operator_context.prompt_for_context`. A future regression in the actual
source (wrong variable, wrong condition, deleted guard) would not be caught by the behavioral
tests — only by the structural `inspect.getsource` checks, which are fragile string searches.

**Fix:** Replace the inline helpers with direct calls to the source logic. Since the functions
involve interactive `input()` calls, use `unittest.mock.patch('builtins.input', return_value=raw)`:

```python
def _parse_years_interactive(raw_input: str) -> int:
    with patch("builtins.input", return_value=raw_input):
        # Only exercise the years block; short-circuit the rest via side_effect
        # or call a dedicated helper extracted from interactive_config.
        ...
```

Alternatively, extract the years-parsing block into a standalone helper in each module
(`_parse_years(raw: str) -> int`) that can be imported and tested directly.

### WR-02: `years_raw` reference outside the `try` block is a static-analysis false negative waiting to happen

**File:** `quirk/interactive.py:203`

**Issue:** `years_raw` is assigned inside the `try` block (line 197). The `except` handler on
line 199 sets `data_longevity_years = 7` without assigning `years_raw`. If `input()` raises
`EOFError`, `years_raw` is never bound; control reaches line 201, where `7 < 1` is `False`,
so line 203 is never reached — no `NameError` at runtime. However:

- Any linter that performs unbound-variable analysis (ruff, pyflakes) will flag `years_raw` on
  line 203 as "possibly undefined", generating a false positive that obscures real issues.
- If the default is ever changed from `7` to a value `< 1` (e.g., during refactoring), a
  `NameError` would surface at runtime for the EOFError path.

**Fix:** Assign a sentinel before the `try` block:

```python
years_raw = ""
try:
    years_raw = input("Data longevity years [7]: ").strip()
    data_longevity_years = int(years_raw) if years_raw else 7
except (ValueError, EOFError):
    data_longevity_years = 7
if data_longevity_years < 1:
    raise ValueError(
        f"Data longevity years must be at least 1 (got: {years_raw!r})"
    )
```

### WR-03: `operator_context.py` uses `except Exception` (too broad) — inconsistent with `interactive.py`

**File:** `quirk/assessment/operator_context.py:41`

**Issue:** The years-parsing block in `prompt_for_context` catches `except Exception` (line 41),
while the parallel block in `interactive.py` correctly catches only `(ValueError, EOFError)`.
`except Exception` silently swallows any exception from `input()` or `int()`, including
programming errors (e.g., `AttributeError` from a botched refactor of the assignment). This
masks bugs and is inconsistent with the companion function.

**Fix:** Narrow to the same exception set used in `interactive.py`:

```python
except (ValueError, EOFError):
    years = 7
```

### WR-04: `test_all_ensure_functions_idempotent` has a fragile signature assumption

**File:** `tests/test_init_db_idempotent.py:44-57`

**Issue:** The test discovers all `_ensure_*` functions via `dir(quirk_db_module)` and calls
each with a single `engine` argument (lines 56-57). The only exclusion is
`_ensure_parent_dir` (which takes a path string). If a future `_ensure_*` helper is added
with a different signature — for example, `_ensure_phase_X_columns(engine, extra_param)` — the
test will raise `TypeError` rather than a meaningful assertion failure. The `>= 5` floor
assertion also won't catch a new function with the wrong signature until the test is run.

**Fix:** Add a signature-guard assertion inside the discovery loop, or switch to an explicit
allowlist of known functions so new additions must be consciously included:

```python
import inspect
for fn in ensure_funcs:
    sig = inspect.signature(fn)
    assert list(sig.parameters.keys()) == ["engine"], (
        f"{fn.__name__} signature changed — update test or exclusion list"
    )
    fn(engine)
    fn(engine)
```

### IN-02: `test_uses_date_arithmetic_not_lex` is a structural test coupled to source layout

**File:** `tests/test_compliance_status_staleness.py:64-93`

**Issue:** The test uses `inspect.getsource(mod.status_report)` to assert the string
`"datetime.date.fromisoformat"` is present in the source. This is fragile: if the function is
moved, wrapped, or if the import alias changes (e.g., `from datetime import date` + `date.fromisoformat`),
the structural assertion fails even though the behavior is correct. The behavioral test in the
same function (`test_keeps_oldest_last_verified`) already covers the correctness; the structural
test adds little value and will produce confusing failures on refactors.

**Fix:** Remove the `inspect.getsource` assertion and rely on the behavioral tests alone, or
replace with a direct invocation test using a case where lex sort and date sort differ
(e.g., `"2026-2-1"` vs `"2026-01-10"`).

### IN-03: Misleading "microsecond precision" in comments and test file

**File:** `quirk/dashboard/api/routes/trends.py:7,44`, `quirk/intelligence/trends.py:6,8,84,86`, `tests/test_trends_subsecond_sessions.py:70`

**Issue:** Multiple comments and docstrings claim the session grouping uses "microsecond
precision". SQLite's `strftime('%f')` produces only 3 decimal digits (milliseconds). The test
comment at line 70 explicitly states: `"# strftime %f produces 6-digit microseconds in SQLite"`
— this is factually incorrect. The actual granularity is millisecond.

**Fix:** Replace "microsecond" with "millisecond" in all affected docstrings and comments.
Update the test comment at line 70:

```python
# strftime %f produces 3-digit milliseconds in SQLite (not 6-digit microseconds)
```
