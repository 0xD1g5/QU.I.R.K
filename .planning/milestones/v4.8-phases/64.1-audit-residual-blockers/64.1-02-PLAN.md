    **The 14 dispositions (apply these decisions; cite the specific audit text for each rationale):**

    1. **scanners-protocol/CR-07** — Nested ThreadPoolExecutor + sslyze pool leak
       - Status: `[ ] deferred-v4.9`
       - Rationale: sslyze invocation in tls_scanner uses a context-manager pool nested inside the scanner's own ThreadPoolExecutor; on exception inside sslyze's pool, the inner pool can leak threads.
       - Safe to defer: Triggers only on sslyze internal exception during TLS enumeration. Resource leak is bounded by scan lifetime (single-shot CLI invocation; FastAPI dashboard scans are also bounded). No long-running daemon.
       - Fix phase: v4.9 (scanner reliability sweep — would naturally pair with WR-12 thread pool sizing)
       - Risk: low — daemon mode does not exist; per-scan leak is reclaimed at process exit

    2. **scanners-protocol/CR-08** — fingerprint._tcp_connect socket leak on SSH banner branch
       - Status: `[ ] deferred-v4.9`
       - Rationale: SSH banner branch in fingerprint._tcp_connect opens a socket but on early-return error path does not always close it.
       - Safe to defer: Sockets are reclaimed at scan-process exit; SSH fingerprinting is a niche code path; no production deployments report descriptor exhaustion.
       - Fix phase: v4.9 scanner reliability sweep
       - Risk: low — bounded by single scan run

    3. **scanners-cloud/CR-01** — migration_planner.py is a 16-line stub
       - Status: `[ ] wont-fix`
       - Acceptable because: migration_planner is dead code — the real migration recommendation logic lives in quirk/intelligence/migration_advisor.py (and was audited there as qramm-compliance/WR-09). The stub will be removed in v5.2 chaos-lab + tech-debt sweep per AUDIT-SUMMARY recommendation. No code path imports the stub today.
       - Fix phase: v5.2 dead-code removal (delete file)
       - Risk: low — unused module; misleading filename only

    4. **scanners-cloud/CR-02** — GCP Cloud SQL stuffs severity into cert_pubkey_alg field
       - Status: `[ ] deferred-v4.9`
       - Rationale: gcp_connector._scan_cloud_sql writes the severity string ("WARN", "CRITICAL") into the cert_pubkey_alg column, conflating algorithm and severity.
       - Safe to defer: Output is consumed only by GCP-specific reporting; users see the conflation in raw JSON but the dashboard's severity badge is derived independently from the severity column. No incorrect findings; just data smell.
       - Fix phase: v4.9 GCP polish
       - Risk: low — cosmetic field misuse; severity is still computed correctly elsewhere

    5. **scanners-cloud/CR-03** — K8s scan_k8s_targets calls _scan_aks_encryption with None cred
       - Status: `[ ] deferred-v4.9`
       - Rationale: When azure_cred is unavailable but AKS targets are present, _scan_aks_encryption is invoked with None and fails with AttributeError caught by a bare except.
       - Safe to defer: K8s is opt-in via the cloud profile + extras install; most users hit the documented "azure-identity not installed" path long before this branch. Failure is silent (caught) — no incorrect findings, just missing data.
       - Fix phase: v4.9 K8s reliability (paired with CR-09 K8S-03 empty-list edge case)
       - Risk: medium — silent data loss for AKS users who have credentials misconfigured

    6. **scanners-cloud/CR-06** — Cache TTL boundary inverted on ttl_hours <= 0
       - Status: `[ ] deferred-v4.9`
       - Rationale: Cache._is_fresh treats ttl_hours <= 0 as "always fresh" instead of "never fresh"; documentation says ttl=0 disables caching.
       - Safe to defer: Default ttl_hours is 24; no production config sets ttl <= 0. Operators who want to disable cache use --no-cache or set ttl very low (e.g., 0.001h) which still works correctly.
       - Fix phase: v4.9 cache subsystem cleanup
       - Risk: low — only manifests when operator deliberately sets ttl <= 0

    7. **scanners-cloud/CR-07** — TokenBucket starvation when tokens > capacity requested
       - Status: `[ ] deferred-v4.9`
       - Rationale: TokenBucket.acquire(n) where n > capacity loops forever because the bucket can never accumulate that many tokens.
       - Safe to defer: All current callers request n=1 (single API call). No code path requests n > capacity. Defensive precondition could be added but no operational risk today.
       - Fix phase: v4.9 rate-limiting sweep (paired with CR-08 contention starvation)
       - Risk: low — no current caller triggers; only manifests if future code requests n > capacity

    8. **scanners-cloud/CR-08** — TokenBucket sleep + busy-wait contention starvation
       - Status: `[ ] deferred-v4.9`
       - Rationale: Under high concurrent acquire load, the sleep-then-retry loop without a fair queue can starve some callers.
       - Safe to defer: Cloud scanners run with ThreadPoolExecutor max_workers=8 typically; bucket sizing is per-cloud-provider rate limits. Real-world contention is bounded. No production starvation reports.
       - Fix phase: v4.9 rate-limiting sweep
       - Risk: medium — could manifest under aggressive parallelism settings

    9. **scanners-cloud/CR-09** — K8s scan_k8s_targets empty-list edge case violates K8S-03
       - Status: `[ ] deferred-v4.9`
       - Rationale: When target list is empty after filtering, scan_k8s_targets returns a malformed empty result that violates the K8S-03 contract (expected: empty list with metadata; actual: bare empty list).
       - Safe to defer: Empty-target path is exercised only when all K8s clusters are explicitly excluded. Dashboard handles bare empty list gracefully.
       - Fix phase: v4.9 K8s reliability (pair with CR-03)
       - Risk: low — schema cosmetic; no user-visible incorrect data

    10. **scanners-cloud/CR-10** — Azure Blob key_source microsoft.storage conflated with absent
        - Status: `[ ] deferred-v4.9`
        - Rationale: azure_blob._classify treats key_source="Microsoft.Storage" and key_source=None as the same case, masking the distinction between platform-managed-keys and unknown-state.
        - Safe to defer: Both states produce the same finding severity (WARN — not customer-managed); the conflation only affects the detail string, not classification.
        - Fix phase: v4.9 Azure polish
        - Risk: low — finding severity unchanged; only diagnostic text affected

    11. **api-cli-core/CR-04** — QRAMMProfile.session_id nullable, no DB-level FK
        - Status: `[ ] deferred-v4.9`
        - Rationale: QRAMMProfile model declares session_id as nullable=True with no SQLAlchemy ForeignKey constraint; orphan profiles can exist after session deletion.
        - Safe to defer: delete_session does cascade in application code (Phase 51 D-09); the missing DB-level FK is a defense-in-depth gap, not a current bug.
        - Fix phase: v4.9 QRAMM data model hardening (pair with CR-05)
        - Risk: low — application-level cascade works; DB-level constraint is belt-and-suspenders

    12. **api-cli-core/CR-05** — delete_session does not clear qramm_sessions.profile_id link
        - Status: `[ ] deferred-v4.9`
        - Rationale: delete_session removes QRAMMSession but does not null out the reverse profile_id pointer on QRAMMProfile, leaving the profile with a stale FK string.
        - Safe to defer: No code path reads profile.session_id without re-validating against the live sessions table; the stale pointer is inert.
        - Fix phase: v4.9 QRAMM data model hardening (pair with CR-04)
        - Risk: low — dangling pointer is read-only stale data

    13. **api-cli-core/CR-06** — Bare except: pass in classifier call drops findings silently
        - Status: `[ ] deferred-v4.9`
        - Rationale: One classifier invocation path swallows all exceptions with bare `except: pass`, dropping findings without logging.
        - Safe to defer: Classifier exceptions are extremely rare (input is structured Pydantic models); failure mode is silent under-count, not crash. No false data emitted.
        - Fix phase: v4.9 logging-hygiene sweep (would naturally pair with the bare-except findings across WR-03, WR-08, IN-02)
        - Risk: medium — silent finding loss is invisible to operators

    14. **api-cli-core/CR-07** — SQL injection guard on column names lacks col_type DDL fragment
        - Status: `[ ] deferred-v4.9`
        - Rationale: _SAFE_COL_RE allowlist validates column names but the col_type DDL fragment (e.g., "TEXT", "VARCHAR(16)") is interpolated without separate validation.
        - Safe to defer: col_type values are hardcoded constants in _V43_COLUMN_DDLS, _PHASE41_COLUMN_DDLS, etc. — never user-influenced. No injection surface today; the gap is "future contributor adds a dynamic col_type" defense.
        - Fix phase: v4.9 schema-migration safety sweep (pair with v4.9 init_db idempotency hardening that Phase 64.1's test will protect)
        - Risk: low — current attack surface is zero (all col_type values are repo constants)
