  <how-to-verify>
    Prereqs:
    1. Ensure `QUIRK_API_TOKEN` is set in the environment used by `quirk serve` (Phase 58 single-user auth).
    2. Build the dashboard: `cd src/dashboard && npm run build`
    3. Start the API: `quirk serve` (or the project's equivalent dev command)
    4. Open the dashboard URL in a browser; authenticate with the bearer token.

    **Case A — Form render (UI-SCAN-01)**

    1. Click "New Scan" in the sidebar (accent-colored button at top, above all nav items)
    2. Verify the URL changes to `/scan/new`
    3. Verify the form shows EXACTLY 4 controls:
       - Targets textarea (empty, monospaced, with placeholder "192.168.1.0/24, api.example.com")
       - Profile radio group with Quick / Standard / Deep — Standard is selected by default
       - Calibration radio group with Strict / Balanced / Lenient — Balanced is selected by default
       - "Enable nmap discovery" checkbox — UNCHECKED by default
    4. Try to submit with empty targets; verify inline error "Targets field is required." appears
    5. Type `@/tmp/x.txt` and submit; verify inline error mentions "@file paths are not supported from the dashboard"

    **Case B — Live stage progression (UI-SCAN-02)**

    1. On /scan/new, enter a quick scan target (e.g. `127.0.0.1`)
    2. Leave profile=Standard, calibration=Balanced, nmap=off
    3. Click "Run Scan"
    4. Verify URL changes to `/scan/job/<uuid>` within ~1 second
    5. Verify the page renders:
       - "Scan Progress" heading + "Running" badge (orange/high)
       - 7-step stage indicator
       - Progress bar starting near 0
       - Job ID in monospace
       - "Cancel scan" destructive (red) button at the bottom
    6. Observe the page for ~30-90 seconds: verify the stage label updates from "Stage 1 of 7 — Discovery" through "Stage 7 of 7 — Reports", with the dot for each completed stage turning teal and the current stage pulsing accent

    **Case C — Post-completion navigation (UI-SCAN-03)**

    1. Continuing from Case B, wait for the scan to complete
    2. Verify the page auto-navigates to `/` (executive summary)
    3. Verify the scan switcher at the top shows the new scan selected (the just-completed scan_run_id)
    4. Verify the executive summary displays the readiness score and findings of the new scan — the scan is indistinguishable from a CLI-launched scan

    **Case D — Cancel button stops the subprocess (UI-SCAN-03)**

    1. From the sidebar, click "New Scan" again
    2. Submit a long-running scan (e.g. a `/24` CIDR like `192.168.1.0/24` with profile=Deep)
    3. On /scan/job/:jobId during a stage other than Reports, click "Cancel scan"
    4. Verify the page navigates to /scan/new immediately
    5. In a separate terminal, run `ps aux | grep run_scan` — verify NO `run_scan.py` process with the cancelled scan's pid remains (cancelled within seconds)
    6. Open the SQLite DB and `SELECT status, error_message, completed_at FROM scan_jobs WHERE job_id = '<the cancelled id>';` — verify status='cancelled' and completed_at is set

    After all four cases pass, record results in `.planning/phases/65-dashboard-initiated-scan/65-HUMAN-UAT.md` using the project template (one row per case with `[x] passed` or `[ ] failed: <reason>`).
