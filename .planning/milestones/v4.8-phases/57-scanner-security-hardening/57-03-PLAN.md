<threat_model>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| network → JWT scanner | JWKS fetches travel over TLS. An on-path attacker can substitute keys if TLS verification is disabled. |
| operator config → JWT scanner runtime | SecurityCfg.allow_insecure_jwks gates whether unverified fetches are permitted. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-57-07 | Tampering | _fetch_jwks httpx.get call | mitigate | Replace `verify=False` literals with `verify=verify_tls` (default True). Operator must explicitly opt out via SecurityCfg.allow_insecure_jwks; an unverified fetch records a HIGH advisory CryptoEndpoint per JWKS URL so the consulting deliverable surfaces the exact unsafe targets. Closes CR-01. |
| T-57-08 | Spoofing | JWKS key inventory | mitigate | Default verify=True ensures harvested keys are TLS-authenticated before being recorded as evidence in the CBOM. |
| T-57-09 | Information Disclosure | Advisory record content | accept | Advisory uses `host=<jwks_url>` which may include the issuer FQDN. Acceptable: this is exactly the data needed for operator triage; no credentials are stored. |

**Residual risks:**
- TLS verification only validates the chain — it does NOT pin against a known
  JWKS key set. A compromised CA can still issue a valid cert for the issuer
  domain. Out of scope for v4.8; would require certificate pinning per issuer.
- The two-fetch flow (well-known + openid-configuration discovery) emits two
  advisories under opt-out. This is intentional (D-10 — "advisories fire once
  per affected target") so operators see both URLs.
