<objective>
Resolve DEF-999.83-C (gitea source seeding 409 duplicate-org/duplicate-repo on
re-run against a persisted `gitea_data` volume).

The current `source/seed.sh` already has a per-repo `repo_exists` short-circuit
(lines 22-26) and uses `|| true` on file PUTs to swallow 409. What it does NOT
have is a TOP-LEVEL org-existence check that short-circuits the whole seeding
pass on a fully-seeded volume — this plan adds it.

Note on the org identity: the user prompt mentions `/api/v1/orgs/quirk-lab` as
the canonical org name. Reading the existing `seed.sh`, the script provisions
repos under the `labadmin` user (`/api/v1/repos/${ADMIN_USER}/...`), not under
an org. Executor: inspect the seed script end-to-end and choose the correct
short-circuit predicate:
  - If the script uses an org (e.g. `quirk-lab`), check `/api/v1/orgs/quirk-lab`
  - If the script uses user repos under `labadmin`, check
    `/api/v1/users/labadmin/repos` returning a NON-EMPTY list (or check a
    canonical sentinel repo like `crypto-antipatterns-python`)
Pick whichever matches the actual seed.sh behavior and document the choice
in SUMMARY.

Purpose: Close ROADMAP Phase 82 success criterion #3 — re-up on a seeded volume
is a no-op with seed container in `Exited (0)`.

Output: `source/seed.sh` with early existence-check short-circuit; gitea-seed
service block pinned in docker-compose.yml; expected_results_v4.md source
section updated.
</objective>
