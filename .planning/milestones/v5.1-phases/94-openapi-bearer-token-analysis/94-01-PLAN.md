<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| operator → analyze-token CLI | Untrusted token text (positional/@file/stdin) crosses into the decoder |
| credential → CBOM builder | Operator-supplied bearer token's declared algorithm crosses into stored CBOM |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-94-01 | Spoofing | analyze_token_cmd alg:none handling | mitigate | Case-insensitive `header["alg"].lower()=="none"` on decoded HEADER dict → CRITICAL + exit 1 (not raw-string search) |
| T-94-02 | Information Disclosure | token value in stdout/logs | mitigate | Token never persisted (no DB write); exceptions wrapped in safe_str(); test asserts token absent from stdout |
| T-94-03 | Information Disclosure | CBOM bearer component | mitigate | Only the declared algorithm is registered; raw token never reaches CryptoEndpoint/CBOM; label exactly `declared_algorithm (unverified)`, never enforced |
| T-94-04 | Tampering | PyJWT decode without algorithms list | mitigate | Always pass `algorithms=[alg]` per PyJWT≥2.4 hardening; opaque tokens caught via DecodeError, INFO + exit 0 |
| T-94-SC | Tampering | npm/pip installs | mitigate | No package installs in this plan (token analysis uses core PyJWT); install gate handled in plan 02 |
</threat_model>
