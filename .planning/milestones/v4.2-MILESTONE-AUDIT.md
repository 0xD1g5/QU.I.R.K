#### Critical Integration Gap — ISSUE-3 (HIGH)

**Scan-window timing causes DNSSEC/SAML endpoints to be excluded from /api/scan/latest**

- **Root cause:** Scanners run sequentially in `run_scan.py` (DNSSEC → SAML → KERBEROS). Each scanner stamps `scanned_at = datetime.now(timezone.utc)` at endpoint creation time. If Kerberos targets are unreachable (default 10s TCP timeout per target × N targets), Kerberos endpoints get stamped 10–20+ seconds after DNSSEC and SAML endpoints.
- **Effect:** `GET /api/scan/latest` anchors its 1-second scan-window on `MAX(scanned_at)`. When Kerberos runs last and times out, `MAX(scanned_at)` is the Kerberos timestamp; the query window `[kerberos_ts, kerberos_ts + 1s)` then excludes all DNSSEC and SAML endpoints that were stamped minutes earlier.
- **Concrete example:** 3 DNSSEC targets + 3 SAML targets scan in 0.5s total. 2 unreachable Kerberos targets × 10s timeout each = +20s lag. Window anchors at T+20s; DNSSEC/SAML endpoints at T+0 to T+0.5s are silently excluded.
- **Note:** This is a timing issue, NOT a timezone format issue. SQLAlchemy strips tzinfo before writing to SQLite regardless.
- **Fix:** Pass a `session_start` timestamp from `run_scan.py` into each scanner; stamp all endpoints from the shared timestamp rather than calling `datetime.now()` at construction time.
- **Affects:** KERB-04, SAML-05, DNSSEC-04, IDENT-02, IDENT-03
- **Severity:** HIGH

#### Integration Gap — ISSUE-2 (MEDIUM)

**`ldap3` not declared in pyproject.toml — KERB-03 LDAP feature is permanently inert**

- **File:** `quirk/scanner/kerberos_scanner.py` `_probe_ldap_anon()`
- **Issue:** `import ldap3` inside `try/except ImportError`, but `ldap3` absent from all `pyproject.toml` groups including `[identity]` extras. `pip install quirk[identity]` will never install ldap3. Graceful-degradation is always taken; no LDAP bind ever occurs.
- **Fix:** Add `ldap3>=3.4` to the `[identity]` extras group in `pyproject.toml`
- **Affects:** KERB-03, INFRA-03
- **Severity:** MEDIUM

#### Integration Gap — NEW-ISSUE-1 (MEDIUM)

**OIDC RS256 findings dropped from identity_findings; appear mislabeled as TLS-sourced in main findings**

- **File:** `quirk/dashboard/api/routes/scan.py` `_derive_identity_findings()` lines 217–256
- **Issue:** Only produces `IdentityFinding` objects for SAML endpoints when `cert_pubkey_alg == "SHA1"` or `cert_pubkey_size < 2048`. OIDC endpoints with RS256 at 2048 bits match neither condition. They fall through to `_derive_findings()` at line 150 where `source="tls"` is hardcoded — so they appear in the Findings tab as TLS-sourced and are absent from the Identity tab entirely.
- **Fix:** Add RS-family algorithm branch to `_derive_identity_findings()` for `protocol="SAML"` endpoints; emit `IdentityFinding(source="saml", ...)` with severity from `OIDC_ALG_SEVERITY`.
- **Affects:** SAML-04, IDENT-02, IDENT-03
- **Severity:** MEDIUM

## Conclusion

**All 25 requirements satisfied** at the code level. Nyquist compliance is now **7/7 COMPLIANT** after phases 17–21 VALIDATION.md files were marked complete.

However, the integration re-check identified 3 open gaps (1 HIGH, 2 MEDIUM) that affect runtime behavior:

1. **ISSUE-3 (HIGH):** Scan-window timing defect — DNSSEC and SAML endpoints may be silently excluded from `GET /api/scan/latest` when Kerberos targets are unreachable and the scan runs sequentially. 2. **NEW-ISSUE-1 (MEDIUM):** OIDC RS256 findings dropped from `identity_findings` — scan.py `_derive_identity_findings()` has no RS-family branch for SAML endpoints; they appear mislabeled as TLS-sourced in the main Findings tab, absent from the Identity tab.
3. **ISSUE-2 (MEDIUM):** `ldap3` absent from `pyproject.toml` — KERB-03 LDAP enumeration is permanently inert for all installations. Fix is one dependency line.

These are candidates for a follow-on gap closure phase before completing the milestone.
