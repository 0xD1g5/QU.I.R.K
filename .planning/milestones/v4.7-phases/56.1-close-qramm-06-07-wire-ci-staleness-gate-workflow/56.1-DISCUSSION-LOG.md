## Q1 — Workflow scope

**Options presented:**
- QRAMM-only (narrow)
- QRAMM + Compliance (Recommended)
- Full pytest suite

**User chose:** QRAMM + Compliance.

**Decision:** D-01. Single workflow file `python-staleness.yml` runs both
test files in one pytest invocation. Generic naming so future
staleness-gated catalogs can be added without renaming.

## Q2 — Trigger model

**Options presented:**
- PR + weekly cron (Recommended)
- PR only
- Cron only (daily)
- PR + daily cron

**User chose:** PR + weekly cron.

**Decision:** D-02. Triggers are `pull_request`, `push: main`, and
`schedule: cron: '0 9 * * 1'` (Mondays 09:00 UTC). Single job runs all
three trigger paths.

## Q3 — Path filters

**Options presented:**
- No path filter (Recommended)
- Filter to model/test paths
- Filter to Python files broadly

**User chose:** No path filter.

**Decision:** D-03. Workflow runs on every PR/push regardless of files
touched. Justification: total runtime <30s; cost of "always run" is
negligible vs cost of a docs-only PR delaying staleness detection.

## Q4 — Cadence documentation home

**Options presented:**
- CLAUDE.md + model_meta docstring (Recommended)
- docs/maintenance.md (new file)
- CONTRIBUTING.md
- All three

**User chose:** CLAUDE.md + model_meta docstring.

**Decision:** D-05. New "Staleness Review Cadence" section in CLAUDE.md;
keep existing docstring in `model_meta.py`. CLAUDE.md is the project's
existing enforcement document for operational invariants.
