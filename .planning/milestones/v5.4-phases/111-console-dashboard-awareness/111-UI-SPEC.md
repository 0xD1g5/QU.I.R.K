## Spacing Scale

Standard 8-point scale — matches the existing codebase `space-y-*` / `gap-*` usage throughout
`data-at-rest.tsx`, `executive.tsx`, and `findings.tsx`. No exceptions for this phase.

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline badge padding |
| sm | 8px | Compact row spacing, inline gap-2 |
| md | 16px | Default element spacing, card padding |
| lg | 24px | Section spacing (space-y-6) |
| xl | 32px | Between major layout blocks (space-y-8) |
| 2xl | 48px | Major section breaks |
| 3xl | 64px | Page-level spacing (unused in this phase) |

Exceptions: min-h-[44px] on all nav links and interactive controls — existing project a11y rule
from sidebar.tsx. The Sensors nav link must also carry `min-h-[44px]`.

## Color

All color values are CSS variable references — hardcoded HSL literals only where CSS variables are
not available (Recharts internals, per existing pattern in `executive.tsx` and `data-at-rest.tsx`).

| Role | Token / Value | Usage |
|------|--------------|-------|
| Dominant (60%) | `hsl(var(--background))` = `220 29% 5%` | Page shell bg |
| Secondary (30%) | `hsl(var(--card))` = `220 21% 7%` / `hsl(var(--secondary))` = `220 18% 11%` | Cards, sidebar, nav, table rows |
| Accent (10%) | `hsl(var(--accent))` = `180 37% 47%` (#4ba8a8 teal) | Reserved list below |
| Destructive | `hsl(var(--destructive))` = `0 68% 60%` | Destructive actions only (none in this phase) |

Accent (`hsl(var(--accent))`) is reserved for:
- Active sidebar nav link highlight (`bg-accent/10`, `border-accent`)
- The new "Sensors" nav link active state (same pattern as all other nav items)
- Focus rings (`ring-ring` = teal)

**Status badge color mapping** (sensor registry — see sensor badge section below):
- `current` (green): `bg-[hsl(var(--quantum-safe))] text-white` — same as "Safe" quantum badge
- `stale` (amber): `bg-[#d4893a]/10 text-[#d4893a] border border-[#d4893a]/28` — matches existing "Partial" pattern in `executive.tsx`
- `unknown` (grey): `bg-secondary text-secondary-foreground` — matches `variant="secondary"` badge

**Coverage warning banner color:**
- Background: `var(--ds-high-dim)` = `rgba(212, 137, 58, 0.10)` (amber tinted, same as stale)
- Border: `var(--ds-high-bdr)` = `rgba(212, 137, 58, 0.28)`
- Icon color: `#d4893a` (amber, same as severity HIGH)
- Text: `hsl(var(--foreground))` for heading, `hsl(var(--muted-foreground))` for body

Score gauge colors are entirely driven by the existing `_gaugeColor()` function — do not override.

### 1. Sensors Page (`/sensors`)

**Route:** `/sensors` — added to `App.tsx` Routes and `NAV_ITEMS` in `sidebar.tsx`.

**Nav placement:** After "Scan History" (`/scans`) and before "Schedules" (`/schedules`) in `NAV_ITEMS`.
Icon: `Radio` from lucide-react (represents networked sensor nodes).

**Page structure** (mirrors `data-at-rest.tsx` pattern):

```
<div className="space-y-6">
  <h1 style={{ fontSize: 20, fontWeight: 600 }}>Sensors</h1>

  [Loading skeleton — 1 heading + 5 table rows using <Skeleton>]
  [Error — <p className="text-muted-foreground text-sm">{error}</p>]

  <Card>
    <CardContent className="p-0">
      <Table> ... </Table>
    </CardContent>
  </Card>
</div>
```

**Sensor registry table columns** (in order):

| Column | Header text | Content | Notes |
|--------|-------------|---------|-------|
| 1 | Sensor ID | `sensor_id` | Monospace: `<span className="font-mono text-xs">` |
| 2 | Segment | `segment` | Plain text, `text-sm` |
| 3 | Version | `sensor_version` | Plain text, `text-sm`; `—` if null |
| 4 | Last Seen | `last_push_at` | ISO → human-readable relative time ("2 hours ago") or "Never" if null |
| 5 | Status | Computed badge | See badge spec below |

**Column header style:** `<TableHead scope="col" className="text-xs font-semibold">` — matches `data-at-rest.tsx` exactly.
**Row hover:** `className="hover:bg-accent/5"` — matches `data-at-rest.tsx` exactly.

**Sensor status badge spec:**

| Status | Condition | Badge classes | Text label | aria-label |
|--------|-----------|---------------|------------|------------|
| `current` | `last_push_at` within cadence | `bg-[hsl(var(--quantum-safe))] text-white text-xs` | "Current" | `aria-label="status: Current"` |
| `stale` | `last_push_at` exists but overdue (> 2× cadence) | `bg-[#d4893a]/10 text-[#d4893a] border border-[#d4893a]/28 text-xs` | "Stale" | `aria-label="status: Stale"` |
| `unknown` | `last_push_at` is null (never pushed) | `variant="secondary" className="text-xs"` | "Unknown" | `aria-label="status: Unknown"` |

Badge color is NOT the only status signal — text label is always present (accessibility rule).
All three states render text alongside color.

**Empty state** (no sensors enrolled):

```tsx
<EmptyStateCard message="No sensors enrolled. Run: quirk sensor enroll --console <url> to register a sensor." />
```

**Loading skeleton:**
```tsx
<div role="status" aria-label="Loading sensors" className="space-y-6">
  <span className="sr-only">Loading...</span>
  <Skeleton className="h-7 w-28" />
  {Array.from({ length: 5 }).map((_, r) => (
    <Skeleton key={r} className="h-10 w-full" />
  ))}
</div>
```

### 2. Segment Filter Dropdown

**Used on:** `FindingsPage` and `CbomPage` (both pages add the filter independently; they share
the same UX pattern but each page manages its own state).

**Control:** `<Select>` from `@/components/ui/select` — already imported in both pages.

**Placement:** In the existing filter bar at the top of each page, appended after current filters.
- `FindingsPage`: after the `protocolFilter` Select (third filter control in the row).
- `CbomPage`: after the `qsFilter` Select (third filter control in the row).

**Exact markup pattern:**

```tsx
<Select value={segmentFilter} onValueChange={setSegmentFilter}>
  <SelectTrigger className="w-40 h-8 text-sm" aria-label="Filter by segment">
    <SelectValue placeholder="All segments" />
  </SelectTrigger>
  <SelectContent>
    <SelectItem value="all">All segments</SelectItem>
    {distinctSegments.map((seg) => (
      <SelectItem key={seg} value={seg}>{seg}</SelectItem>
    ))}
  </SelectContent>
</Select>
```

**Default state:** `"all"` — maps to no `?segment=` query param on API calls (all segments shown).
**NULL-segment scans:** When `segment` is `null` (single-host / pre-v5.4 scan), those rows appear
in "All segments" view and are invisible in any named-segment filter. This is correct and intended.
**No sensors enrolled:** `distinctSegments` is empty; only the "All segments" option renders — the
control is always present and usable (does not hide when no segments exist).
**Keyboard navigation:** Select uses Radix UI — fully keyboard-navigable by default. No additional
keyboard handling required.

### 3. Per-Segment Score Gauges (Executive / Merged View)

**Location:** Executive Summary page — inside the existing `<Card>` that wraps the score gauges row.

**Data source:** `/api/merge/latest` response (new endpoint). When no merge exists, this data is
null and only the org-wide gauge renders (no per-segment gauges, no change to existing layout).

**Layout — when merge data is present:**

```
<div className="flex flex-wrap justify-around gap-8">
  [Org-wide gauge — isOverall=true, size=160, existing]
  [Per-segment gauge 1 — size=120, maxValue=100]
  [Per-segment gauge 2 — size=120, maxValue=100]
  ...
</div>
```

The existing flex-wrap layout in `executive.tsx` naturally accommodates additional gauges without
CSS changes — the gauges wrap to a second row when width is exceeded.

**Per-segment ScoreGauge props:**

```tsx
<ScoreGauge
  score={segScore}          // 0..100 (Option A: per-segment re-scored through engine)
  label={segmentLabel}      // segment name string, e.g. "dmz" or "corp"
  size={120}
  maxValue={100}            // full 0-100 range, same as org-wide gauge
/>
```

`maxValue={100}` — per-segment scores are re-run through the full scoring engine and produce a
0-100 result, identical to the org-wide scale.

**Recharts mounting rule (project rule, must not be violated):**
All chart children (`<Bar>`, `<Cell>`, etc.) in `executive.tsx` are statically mounted. The
per-segment gauges are SVG-based (not Recharts), so this rule does not apply to the gauges
themselves. Any future Recharts additions in this phase must keep chart children statically
mounted and toggle visibility via `fillOpacity`/`strokeOpacity`.

**Label truncation:** Segment labels are truncated at 16 characters with ellipsis in the gauge
`label` prop: `label.length > 16 ? label.slice(0, 15) + '…' : label`. The gauge's 120px width
accommodates approximately 12-14 characters at 12px/600 weight.

**No-merge graceful state:** When `/api/merge/latest` returns null/empty, no per-segment gauges
render. The org-wide gauge row is visually unchanged. No empty state needed — absence is the
correct display.

### 4. Coverage Warning Banner

**Trigger:** Latest `merge_runs.coverage_warning` is non-null (one or more sensors were missing
at merge time). Parsed from `coverage_warning_json`.

**Placement:** Immediately above the score gauges Card in the Executive Summary page —
same position as `<RegressionAlertChip />`. Renders before the gauges so the consultant sees
the caveat before interpreting the scores.

**Not dismissible.** The coverage warning reflects real structural incompleteness of the merged
score. It must remain visible until the underlying condition resolves (all sensors check in and
a new merge is run). Dismissal is deferred to v5.5. This distinguishes it from `RegressionAlertChip`
which is dismissible because it is informational.

**Visual structure** (matches `RegressionAlertChip` structural pattern, different color/icon):

```tsx
<div
  className="flex items-start gap-3 rounded-md border px-4 py-3 mb-6"
  style={{
    background: "var(--ds-high-dim)",          // rgba(212, 137, 58, 0.10)
    borderColor: "var(--ds-high-bdr)",         // rgba(212, 137, 58, 0.28)
  }}
  role="alert"
  aria-live="polite"
>
  <AlertTriangle className="h-4 w-4 mt-0.5 shrink-0" style={{ color: "#d4893a" }} aria-hidden="true" />
  <div className="flex flex-col gap-1">
    <span className="text-sm font-semibold">Incomplete sensor coverage</span>
    <span className="text-sm text-muted-foreground">
      {missingCount} sensor{missingCount !== 1 ? "s" : ""} did not contribute to this merge.
      Scores may understate risk in uncovered segments.
      {missingSegments.length > 0 && (
        <> Missing: <span className="font-mono text-xs">{missingSegments.join(", ")}</span>.</>
      )}
    </span>
  </div>
</div>
```

**Icon:** `AlertTriangle` from lucide-react (already imported in `RegressionAlertChip`).
**`aria-live="polite"`** — banner updates if a new merge completes while the page is open.
**`role="alert"`** — screen reader announces the incomplete coverage condition.
**No dismiss button** — permanent until condition clears.

**Copy tone:** Factual, non-alarming. Avoids "warning" or "danger" language. Tone example:
"Incomplete sensor coverage — scores may understate risk in uncovered segments."
Does NOT use: "DANGER", "ERROR", "your data is unsafe", exclamation marks.

## Interaction States

### Sensors Page

| State | Trigger | Visual |
|-------|---------|--------|
| Loading | API in-flight | Skeleton (h-7 heading + 5× h-10 rows) |
| Empty | 0 sensors enrolled | `EmptyStateCard` with enroll command |
| Populated | ≥1 sensor | Table with status badges |
| Error | API error | Muted text error message |

### Segment Filter

| State | Trigger | Visual |
|-------|---------|--------|
| Default | Page load | "All segments" selected |
| Filtering | User selects segment | Table/CBOM view filters to matching rows |
| No segments available | No distributed sensors yet | Only "All segments" option; control still visible |

### Coverage Warning Banner

| State | Trigger | Visual |
|-------|---------|--------|
| Hidden | No merge run OR coverage_warning is null | Banner absent |
| Visible | Latest merge has missing sensors | Amber banner above gauges |

### Per-Segment Gauges

| State | Trigger | Visual |
|-------|---------|--------|
| Absent | No merge data | Only org-wide gauge |
| Present | Merge data with ≥1 segment | Org-wide + per-segment gauges in flex-wrap row |
