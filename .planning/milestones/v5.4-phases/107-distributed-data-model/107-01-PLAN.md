## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| DDL string construction → SQLite | Column/index names compiled into ALTER/CREATE statements |
| (future) sensor → console token storage | Schema reserves token_hash; raw token never crosses into persistence |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-107-01 | Tampering | `_V54_SENSOR_COLUMNS` DDL fragments | mitigate | Column types (`TEXT`) flow through `_ensure_columns` `_SAFE_COL_TYPE_RE`/`_SAFE_COL_RE` allowlist; no user input reaches DDL |
| T-107-02 | Tampering | `CREATE INDEX` step in init_db | mitigate | Index name, table, column are hardcoded string literals — zero interpolation, no user-controlled input |
| T-107-03 | Information Disclosure | `sensor_tokens.token_hash` | mitigate | Schema stores `String(64)` SHA-256 hex only (per D-03 / token_cmd.py); raw token has no column — persistence of raw token is structurally impossible |
| T-107-04 | Spoofing/Replay | cross-sensor `payload_id` replay | accept | `payload_id` unique + FK `ON DELETE CASCADE` (D-04): re-enrollment mints a fresh sensor UUID and cascades away old dedup history, so stale payload_ids cannot be reused across sensors. Enforcement endpoint is Phase 109. |
| T-107-SC | Tampering | npm/pip installs | accept | No package installs in this phase — schema-only landing using existing SQLAlchemy. No new dependencies. |
</threat_model>
