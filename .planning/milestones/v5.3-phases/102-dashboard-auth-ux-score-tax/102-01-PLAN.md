<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| operator shell → config.yaml | CLI writes the dashboard token to a local YAML file the operator controls |
| config.yaml → require_auth | the persisted token is later read server-side for request authentication |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-102-01 | Information Disclosure | `secrets.token_urlsafe(32)` token generation | mitigate | Use stdlib CSPRNG (256 bits entropy); never derive from time/PID. Locked decision AUTH-01. |
| T-102-02 | Tampering | `_write_token_to_config` YAML write-back | mitigate | Full-file `yaml.safe_load` → update single key → `yaml.dump` so other config blocks are never clobbered (Pitfall 2). Test test_token_generate_preserves_other_keys is the gate. |
| T-102-03 | Information Disclosure | `quirk token show` echoes full token to terminal | accept | Local single-tenant tool; the operator already controls the terminal. Documented; masking adds friction with no real benefit (open-question A1). |
| T-102-04 | Spoofing | `quirk token show` vs active env-var token confusion | mitigate | `show` reads YAML directly and prints a note when QUIRK_API_TOKEN is set, so the operator is not misled about which token is active. |
| T-102-SC | Tampering | npm/pip installs | mitigate | No new packages — all deps (secrets, yaml, hmac) are stdlib/existing per RESEARCH Package Legitimacy Audit. No slopcheck checkpoint required. |
</threat_model>
