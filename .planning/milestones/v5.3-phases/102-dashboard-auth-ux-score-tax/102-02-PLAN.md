## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser/client → dashboard API | untrusted X-API-Key / bearer credentials cross here |
| route registration → CI gate | new routes must inherit require_auth; the gate enforces this at test time |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-102-05 | Spoofing | token comparison in require_auth | mitigate | hmac.compare_digest (constant-time) for BOTH X-API-Key and bearer paths; no short-circuit string == comparison. |
| T-102-06 | Spoofing | empty X-API-Key header | mitigate | `if x_api_key:` guard prevents compare_digest against an empty string; absent/empty header falls through to bearer path. |
| T-102-07 | Elevation of Privilege | header confusion (X-API-Key vs Bearer) | mitigate | Explicit precedence: X-API-Key checked first; if present it is authoritative (valid→pass, invalid→401) and bearer is never consulted, preventing downgrade ambiguity. |
| T-102-08 | Elevation of Privilege | new route ships without auth | mitigate | test_all_data_routes_have_auth_dependency CI gate fails the build for any non-health route missing require_auth (the route-coverage gate Phase 101 referenced). |
| T-102-09 | Information Disclosure | token in URL/query param | mitigate | Auth accepted via header only; no query-param token path is added. |
| T-102-SC | Tampering | npm/pip installs | mitigate | No new packages (hmac is stdlib). No slopcheck checkpoint required. |
</threat_model>
