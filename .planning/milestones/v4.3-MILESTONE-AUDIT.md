## Integration Findings

### B-1 — OIDC RS-family severity not stored on CryptoEndpoint

**File:** `quirk/scanner/saml_scanner.py` lines 333–343 and 349–358 (`_parse_oidc_discovery`)
**Issue:** `severity = OIDC_ALG_SEVERITY.get(alg)` used as inclusion guard but never assigned (`severity=severity`) in CryptoEndpoint constructor. OIDC SAML endpoints stored with `ep.severity = None`.
**Impact:** LOW — downstream is not broken. `scan.py:_derive_identity_findings()` re-derives severity from `cert_pubkey_alg` via `OIDC_ALG_SEVERITY`. Evidence counters use `service_detail` not `severity`. Total requirements satisfied.
**Affected REQs:** None blocked — SAML-04, IDENT-03 still satisfied via indirect path.
**Fix:** Add `severity=severity` argument to both CryptoEndpoint constructors in `_parse_oidc_discovery`.

### B-2 — SubScores Pydantic model missing `data_at_rest` field

**Files:**
- `quirk/dashboard/api/schemas.py` lines 20–25 — `SubScores` has 4 fields (no `data_at_rest`)
- `quirk/dashboard/api/routes/scan.py` lines 595–600 — `SubScores(...)` constructor has 4 args
- `src/dashboard/src/types/api.ts` lines 1–5 — TypeScript interface missing `data_at_rest`
**Issue:** `scoring.py` returns `"data_at_rest": dar_score` in subscores dict (line 207). Pydantic silently drops it at serialization. The total `score` integer is correct (sums all 5 subscores). The `drivers` list correctly shows data_at_rest impact strings (e.g., "Object storage unencrypted"). Only the subscore *breakdown* display omits data_at_rest.
**Impact:** MEDIUM — Phase 27–30's primary dashboard artifact (the data_at_rest subscore) is invisible in the UI subscore breakdown. Product gap: users see score changes from DAR findings but cannot see which subscore was affected.
**Affected REQs:** No REQUIREMENTS.md requirement is unsatisfied. Phase 27 SC-5 requires the scoring engine, not the dashboard display. Classified as tech debt.
**Fix:** Add `data_at_rest: int = 0` to SubScores (Python + TypeScript); add `data_at_rest=subscores_raw.get("data_at_rest", 0)` to scan.py SubScores constructor.

### W-1 — VAULT falls through TLS else-branch in CBOM Pass 1

**File:** `quirk/cbom/builder.py` line 410
**Issue:** No `elif ep.protocol == "VAULT":` handler in Pass 1. VAULT endpoints fall through to TLS default branch, which registers `cert_pubkey_alg` correctly for transit keys. Passes 2 and 3 correctly include "VAULT" in skip lists.
**Impact:** LOW — currently functional. Fragile under future refactoring.
**Affected REQs:** VAULT-01 satisfied via fallthrough — not blocked.
**Fix:** Add explicit `elif ep.protocol == "VAULT":` branch in Pass 1 that handles transit key algorithm registration (similar to how GCP is handled).

### W-2 — `dat_scan_json` never populated for POSTGRESQL/MYSQL/RDS endpoints

**Files:** `quirk/scanner/db_connector.py` (`scan_pg_targets`, `scan_mysql_targets`); `quirk/scanner/aws_connector.py` (`_scan_rds_encryption`)
**Issue:** All CryptoEndpoint constructors in these functions lack `dat_scan_json=` argument. The column exists (from `_ensure_v43_columns`), but is always NULL for DB rows. Evidence counters work via `service_detail` parsing — scoring unaffected.
**Impact:** LOW — functional requirements satisfied. Missing structured JSON context for DB endpoints breaks the "universal DAR scan result JSON" contract stated in Phase 27 summary.
**Affected REQs:** DB-01/02/03 are satisfied via evidence/scoring paths. Not blocked.
**Fix:** Serialize a structured dict to `dat_scan_json` in each DB scanner (pg ssl status, mysql cipher, RDS encryption type).
