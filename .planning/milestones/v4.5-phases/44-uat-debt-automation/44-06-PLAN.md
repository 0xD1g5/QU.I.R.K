## Per-Scenario Cloud-Only Justification (UAT-02)

D-01/D-02/D-03 classify all Phase 29 scenarios as `cloud-only`. Per UAT-02, this label
must be defensible per scenario — minikube/kind cannot simulate the required cloud-managed
control plane APIs. The 10 pending Phase 29 scenarios collapse to 3 UAT scenarios in
docs/UAT-SERIES.md §Phase 29 (UAT-29-01, UAT-29-02, UAT-29-03), each exercising a distinct
provider-specific control plane API:

| Scenario | Cloud-Managed API Required | Why minikube/kind Cannot Provide It |
|----------|----------------------------|--------------------------------------|
| UAT-29-01 (EKS Encryption + Secret-Type Enumeration) | AWS EKS `DescribeCluster` API returning `encryptionConfig[].provider.keyArn` (KMS key ARN); requires `aws eks update-kubeconfig` and an AWS-managed control plane backed by KMS | minikube/kind run a self-managed control plane with no AWS API surface; there is no `encryptionConfig[].provider.keyArn` field to inspect, and no KMS to bind to. The `service_detail=EKS/encrypted|EKS/unencrypted` ladder is derived solely from EKS control-plane metadata. |
| UAT-29-02 (GKE Encryption + Secret-Type Enumeration) | GCP Container Engine `clusters.get` API returning `databaseEncryption.state` (numeric `== 2` for encrypted); requires `gcloud container clusters get-credentials` and a GCP-managed control plane | minikube/kind do not expose `databaseEncryption.state`. The Pitfall-2 numeric comparison (`== 2`) is meaningful only against a real GKE cluster. There is no GCP control plane to query locally. |
| UAT-29-03 (AKS Encryption + RBAC Degradation) | Azure AKS `ManagedClusters.Get` API returning `securityProfile.azureKeyVaultKms` (Key Vault KMS binding); plus a real Azure AAD service principal with restricted RBAC for the K8S-03 403 leg | minikube/kind cannot provide Azure Key Vault KMS bindings — the `AKS/kv-kms` vs `AKS/platform-managed` distinction does not exist. The RBAC-403 degradation path requires a real Azure AAD service principal, not a local-cluster ServiceAccount. |

**Mock coverage acknowledgment (D-02):** All three scenarios' scanner logic
(`_scan_eks_encryption`, `_scan_gke_encryption`, `_scan_aks_encryption`,
`_enumerate_secret_types`, K8S-03 RBAC-403 degradation) is covered by mock-based unit
tests in `tests/test_k8s_connector.py` (15 tests passing). Mock fixtures simulate the
cloud-API responses that minikube/kind cannot produce.

**Conclusion:** No subset of the 3 UAT-29 scenarios can be ported to a local cluster
without losing the control-plane-API behaviour they were written to validate. Cloud-only
classification is per-scenario defensible, not a blanket label.
</phase_29_cloud_only_justification>
