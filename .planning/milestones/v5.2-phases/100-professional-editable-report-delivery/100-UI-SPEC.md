# Phase 100 — UI Design Contract

> Visual and interaction contract for Phase 100: Professional & Editable Report Delivery.
> Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Scope Clarification

Phase 100 has **no net-new React screens**. The UI surfaces are:

1. **PDF cover page** (`report.html.j2`) — a new first-page cover block preceding the existing
   Executive Summary section. Rendered by headless Chromium (JS disabled, offline, A4, 15/12mm
   margins). Source: FMT-01 / D-01..D-04.
2. **PDF print layout & pagination** (`report.html.j2` `<style>` block) — `@page` rules,
   `break-inside`/`break-after` guards, fixed-table-layout for the 7-column findings table,
   `thead` repeat. Source: FMT-02 / D-05..D-08.
3. **DOCX export** (`quirk/reports/docx_renderer.py`, new module) — structural Word document
   derived from the same `ExecContent` / findings / `IntelligenceReport` content model. Source:
   FMT-03 / D-09..D-12.
4. **React dashboard** — NO changes. No new components, no layout changes.
5. **CLI markdown** — NO changes (content already ships Phases 98/99).

All design decisions below apply to the HTML/PDF template and DOCX surfaces unless stated.

## Interaction Contract

### HTML/PDF (static document — no JS interactions)

All Phase 100 additions are static HTML/CSS. No JavaScript, no interactive elements added.

**Cover page block** — A new `<div class="cover-page">` inserted as the first child of
`<div class="report-body">`, before `<section id="executive-summary">`. The cover page
occupies exactly one printed page (`break-after: page`). Chromium renders the remaining
sections starting on page 2.

**Print @page rules** — Inside the `<style>` block:
```css
@media print {
  @page { size: A4; margin: 15mm 12mm; }
  /* Playwright already passes these margins — the @page rule is a belt-and-braces
     declaration ensuring consistency if the HTML is printed directly from a browser. */
  .cover-page { break-after: page; }
  h1 { break-after: avoid; }
  h2 { break-after: avoid; }
  tr { break-inside: avoid; }
  thead { display: table-header-group; }
}
```

**Cover page absent in screen view** — The `.cover-page` block is visible in browser screen
view as the top of the report. It does not disrupt HTML browsing because it has no
fixed/absolute positioning. A consultant opening the HTML in a browser sees the cover, then
scrolls to the report body.

### DOCX (file artifact — no interaction)

The DOCX is written to `{outdir}/report-{stamp}.docx` by `write_reports` in `writer.py`,
after the PDF step. No user interaction. Graceful degradation if python-docx is absent
(advisory to stderr, continues).

### React Dashboard (no interaction changes)

No new interactive elements, no new components, no layout changes.
