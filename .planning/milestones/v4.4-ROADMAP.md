# Milestone v4.4: Data in Motion

**Status:** In Progress
**Phases:** 32–37
**Version Target:** 4.4.0
**Total Plans:** TBD (estimated 18–22)

## Overview

Extends QU.I.R.K.'s cryptographic inventory to cover network transport layers — the protocols
that move data between services. Adds two new scanner surfaces: email protocol TLS (SMTP,
IMAP, POP3 — all 7 standard ports) and message broker TLS (Kafka, RabbitMQ, Redis). Both
surfaces produce findings, evidence counters, CBOM components, and a shared `data_in_motion`
subscore that becomes the 6th pillar of the quantum-readiness score alongside TLS, SSH, API,
Identity, and Data at Rest.

**Surfaces covered:**
- Email: SMTP-STARTTLS (25, 587), SMTPS (465), IMAP-STARTTLS (143), IMAPS (993), POP3-STARTTLS (110), POP3S (995)
- Brokers: Kafka (9092/9093/9094), RabbitMQ/AMQP (5671/5672), Redis (6379/6380), Azure Service Bus (5671), AWS SQS (443)

**Structural carry-forwards from v4.3:**
- STRUCT-01: All new scanners accept `session_start` parameter (shared `datetime` from `run_scan.py`)
- STRUCT-02: All new `[motion]` extras group entries declared in `pyproject.toml` at plan time
- STRUCT-03: Each phase plan includes a `pyproject.toml` diff as a required deliverable if any dependencies change

## Phase Details

### Phase 32: Email Scanner

**Goal**: QU.I.R.K. can audit TLS posture on all seven standard email protocol ports — detecting weak ciphers, legacy TLS versions, expired certificates, and STARTTLS downgrade risk — stored in `email_scan_json` and producing `CryptoEndpoint` rows with correct `ep.protocol` labels
**Depends on**: Phase 31 (v4.3 complete)
**Complexity**: L
**Requirements**: STRUCT-01, STRUCT-02, STRUCT-03, EMAIL-00, EMAIL-01, EMAIL-02, EMAIL-03, EMAIL-04, EMAIL-05, EMAIL-06, EMAIL-07, EMAIL-08, EMAIL-09, EMAIL-10, EMAIL-11, EMAIL-12
**Success Criteria** (what must be TRUE):
  1. Scanning a mail server on any of the 7 default email ports returns TLS version, negotiated cipher suite, cert subject/issuer/expiry, and key algorithm — accessible in the DB `email_scan_json` column and in the scan's `CryptoEndpoint` rows
  2. Port 25 STARTTLS endpoints emit a static MEDIUM `starttls-downgrade-risk` finding regardless of cipher strength; weak RSA key-exchange ciphers (`TLS_RSA_WITH_*`, 3DES, RC4) on any email port emit HIGH findings
  3. A `CONNECTION_REFUSED` on port 25 (commonly blocked on cloud VMs) does not crash the scan — it is handled gracefully and logged
  4. When sslyze fails, the stdlib fallback (`smtplib`/`imaplib`/`poplib`) negotiates STARTTLS and extracts TLS version, cipher, and cert via the underlying `SSLSocket`
  5. `docker compose --profile email up` starts the Postfix+Dovecot container with weak TLS (TLS 1.1 minimum, RSA non-PFS ciphers, self-signed RSA-2048 cert); scanning port 30587 produces at minimum one HIGH finding for weak cipher and one MEDIUM for STARTTLS port-25 equivalent; `labs/email/expected_results.md` documents these
  6. `ep.service_detail` format is `"SMTP-STARTTLS:587"`, `"SMTPS:465"`, `"IMAPS:993"`, `"POP3S:995"` etc. — matching EMAIL-10 convention

### Phase 33: Broker Scanner

**Goal**: QU.I.R.K. can audit TLS posture on Kafka, RabbitMQ, Redis, Azure Service Bus, and AWS SQS — detecting plaintext listeners, weak TLS versions, and quantum-unsafe ciphers — in a single `broker_scanner.py` module following the `db_connector.py` architecture pattern
**Depends on**: Phase 31 (v4.3 complete); can develop in parallel with Phase 32
**Complexity**: L
**Requirements**: STRUCT-01, STRUCT-02, STRUCT-03, BROKER-00, BROKER-ARCH, KAFKA-01, KAFKA-02, KAFKA-03, KAFKA-04, RABBIT-01, RABBIT-02, RABBIT-03, RABBIT-04, RABBIT-05, REDIS-01, REDIS-02, REDIS-03, BROKER-LAB-01, BROKER-LAB-02
**Success Criteria** (what must be TRUE):
  1. Scanning Kafka on port 9093 returns cert chain, accepted cipher suites, and TLS version; scanning plaintext port 9092 emits a HIGH finding (`kafka-plaintext-listener`) when the port responds
  2. Scanning RabbitMQ on port 5671 (AMQPS) via sslyze returns the same data as a TLS port 443 scan (direct TLS, no STARTTLS); plaintext port 5672 with a raw AMQP header probe emits HIGH (`amqp-plaintext-listener`) if the port responds with an AMQP frame
  3. Scanning Redis on port 6380 via raw `ssl.SSLContext` socket wrap returns TLS version, negotiated cipher, and cert; plaintext port 6379 emits HIGH (`redis-plaintext-no-auth`) if Redis responds without authentication; `redis-py` `CONFIG GET tls-*` is attempted as enrichment only and degrades gracefully on `NOAUTH`/`NOPERM`
  4. `docker compose --profile broker up` starts all three broker containers; scanning the chaos lab ports (26379/26380 Redis, 29092/29093 Kafka, 25672/25671 RabbitMQ) produces at minimum plaintext HIGH for all three brokers and weak cipher HIGH for at least two; `labs/broker/expected_results.md` documents these expected findings
  5. All three scanner functions (`scan_kafka_targets`, `scan_rabbitmq_targets`, `scan_redis_targets`) accept `session_start` parameter and stamp `ep.scanned_at` with it — no per-scanner `datetime.now()` calls
  6. `broker_scan_json` column is present in the SQLite schema after first scan; `broker_scanner.py` is a single module exposing three public functions

## Risk Register

### Risk 1: OpenSSL TLS 1.0/1.1 Disabled by Default on Modern Hosts

**Likelihood:** HIGH — documented in the existing chaos lab (same constraint applies to `tls-legacy` nginx scenario)
**Impact:** MEDIUM — chaos lab weak-TLS scenarios may not produce TLS 1.1 findings when scanner runs on macOS 13+ or Ubuntu 22.04+ (OpenSSL 3.x)

**Details:** OpenSSL 3.0+ disables TLS 1.0 and 1.1 at the library level. Postfix/Dovecot containers configured with `ssl_protocols = !SSLv2 !SSLv3` (allowing TLS 1.0/1.1) will still negotiate TLS 1.2 minimum when the connecting client's OpenSSL disallows older versions. The scanner may probe for legacy TLS versions and receive a "no protocols available" error rather than a version-constrained handshake result.

**Mitigations:**
- Target chaos lab weak-TLS scenarios at RSA key exchange (non-PFS ciphers) and weak key sizes as the primary detectable finding, since these are observable at TLS 1.2
- Document the OpenSSL 3.x caveat in `labs/email/expected_results.md` and `labs/broker/expected_results.md` — findings may be cipher-based rather than protocol-version-based depending on scanner host OS
- Accept TLS 1.2 + 3DES/RSA-key-exchange as the canonical "weak TLS" chaos lab baseline (same approach used in existing broker research notes)

---

### Risk 2: sslyze STARTTLS Timeout on Port 25 (Cloud VM Blocking)

**Likelihood:** MEDIUM — port 25 is routinely blocked by cloud providers (AWS EC2, GCP Compute, Azure VM) at the network egress layer
**Impact:** LOW — scanner gracefully handles `CONNECTION_REFUSED`; port 25 findings are supplemental (port 587 is the primary STARTTLS validation port)

**Details:** Cloud VMs commonly block outbound TCP port 25 to prevent spam. A scan that includes `mail.example.com` from an EC2 instance will receive `CONNECTION_REFUSED` or a timeout on port 25 even when the target mail server has STARTTLS configured. This is an infrastructure constraint, not a scanner bug, but it could cause scan duration to spike due to sslyze timeout waits.

**Mitigations:**
- Apply the existing `_categorize_tls_error()` pattern: `CONNECTION_REFUSED` on port 25 is silently skipped with a debug log, not a scan_error finding
- Use `network_timeout` parameter in `ServerNetworkConfiguration` to cap individual port probe time (consistent with existing scanner timeouts)
- Chaos lab routes port 25 to port 30025 on localhost — not subject to cloud egress filtering during development validation

---

### Risk 3: Kafka AdminClient API (KAFKA-04) Auth Complexity

**Likelihood:** HIGH — most production Kafka clusters enforce SASL auth and/or mTLS for AdminClient connections
**Impact:** LOW — KAFKA-04 is explicitly marked `(optional)` in requirements; TLS probe via sslyze (KAFKA-01/02/03) is the required path and has no auth dependency

**Details:** `AdminClient.describe_configs()` via `kafka-python` requires a Kafka connection at the application protocol level. Production Kafka brokers with SASL_SSL listeners will reject AdminClient connections that lack SASL credentials. This means the "enrichment" path is blocked on almost every real-world target. The scanner must not treat AdminClient failure as a scan_error.

**Mitigations:**
- Implement KAFKA-04 with full graceful degradation: `kafka-python` import error → skip silently; AdminClient connection error → log INFO, no scan_error; 403/AUTH failure → log INFO, proceed with sslyze-only results
- Do not promote AdminClient data to a required output field in `broker_scan_json` — it is an optional enrichment key alongside the mandatory sslyze probe results
- Chaos lab Kafka container (bitnami/kafka:3.7 KRaft mode) does not enforce SASL — AdminClient enrichment can be validated against the chaos lab without real cluster credentials
